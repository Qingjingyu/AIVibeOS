# AIVibe OS 团队推行说明

## 目的

这份文档用于说明：AIVibe 现在可以怎么在 团队里正式使用。

它的目标不是做展示，而是让产品、研发、测试在同一套标准下协作。

## 一、AIVibe 当前能解决什么

当前版本已经能解决你们最核心的规范化问题：

- 需求不再只靠口头表达
- 开发前先有 research / proposal / spec / design / tasks
- QA 有独立模板和操作手册
- 项目结束后有 retro 复盘入口

它已经足够作为团队第一版规范入口使用。

## 二、AIVibe 当前不能自动解决什么

AIVibe 不是魔法文档，它不能自动保证：

- 产品经理一定把需求写细
- 开发一定按 spec 做
- QA 一定测透
- 页面一定有质感
- 项目一定一次做对

这些仍然依赖团队执行。

所以正确理解是：

**AIVibe 解决的是“没有标准”的问题，不直接替代“执行质量”。**

## 三、建议怎么在团队里落地

### 1. 产品经理怎么用

新项目开始时，先按下面顺序出文档：

1. `Research`
2. `Proposal`
3. `Spec`

产品经理的核心职责是：

- 把需求写清楚
- 把边界写清楚
- 把页面、流程、状态、文案、验收标准写清楚

### 2. 研发怎么用

研发拿到文档后，不直接开写代码，先做两件事：

1. 审核文档是否足够清楚
2. 基于 `Design` 和 `Tasks` 模板拆方案

研发的核心职责是：

- 补齐技术设计
- 拆清执行步骤
- 按文档实现，而不是边写边猜

### 3. QA 怎么用

QA 在联调后开始工作，必须：

- 先读 `Spec`
- 再读 `Design`
- 再按 `QA 模板` 和 `QA 操作手册`执行

QA 的核心职责是：

- 不只验证“能跑”
- 还要验证“值不值得发”

## 四、建议的执行要求

以后新项目默认执行下面四条：

1. 没有 `Research` 和 `Proposal`，不进入 `Spec`
2. 没有 `Spec`，不允许 AI 或研发直接开工
3. 没有 `QA` 结论，不进入 `Release`
4. 没有 `Retro`，项目不算真正结束

## 五、建议的推行方式

### 第一步：先试点

不要一上来要求所有项目全量执行。

建议先挑 1 个项目完整试跑，优先推荐：

- `AiCore`

### 第二步：边跑边修

试跑过程中记录：

- 哪些模板太空
- 哪些步骤太重
- 哪些地方最容易卡住

然后回写到 AIVibe。

### 第三步：再推广

等试点项目跑完一轮后，再把 AIVibe 作为默认团队规范推广到更多项目。

## 六、给团队的一句话说明

**AIVibe 不是资料库，而是 团队以后做项目的标准入口。**
