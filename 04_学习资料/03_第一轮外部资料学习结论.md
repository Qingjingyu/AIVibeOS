# 第一轮外部资料学习结论

这份文档记录当前第一轮外部资料学习后，对 目标项目 最值得吸收的做法。

结论不是“照搬哪个仓库”，而是把能变成内部标准的部分先抽出来。

## 一、总判断

这批资料对 目标项目 当前最有价值的，不是某个具体插件，而是三类方法：

- `Spec 驱动`
- `Research-first`
- `Workflow 标准化`

你们现在“像 demo、不够生产力、bug 多、返工重”的问题，本质上和这三类方法直接相关。

## 二、最值得吸收的 7 条方法

### 1. 先研究，再写 spec

来源启发：

- `OpenSpec`
- `web-access`
- `everything-claude-code`

吸收建议：

- 新项目默认先做 research
- research 里至少包含：
  - 对标产品
  - 竞品页面
  - 现成方案
  - 视觉参考
  - 业务风险
- research 不做完，不进入正式 spec

## 2. 用固定文档链路代替口头需求

来源启发：

- `OpenSpec`

吸收建议：

每个项目都至少维护四份核心产物：

- `proposal`
- `spec`
- `design`
- `tasks`

这是 目标项目 后续最该固定的最小文档闭环。

## 3. 设计和实现必须分层

来源启发：

- `OpenSpec`
- `superpowers`

吸收建议：

- proposal 决定做不做
- spec 决定做成什么样
- design 决定技术怎么落
- tasks 决定开发怎么拆

不要把这四件事混成一份模糊文档。

## 4. AI 开发必须走标准工作流

来源启发：

- `superpowers`
- `everything-claude-code`
- `gstack`

吸收建议：

AI 开发不再是：

- 给一句话
- 直接开始写

而应该变成：

- brainstorming
- plan
- implementation
- review
- verification
- retro

## 5. 把“设计质感”当成独立输入，而不是开发副产物

来源启发：

- `ui-ux-pro-max-skill`

吸收建议：

- 前端开发前必须先给视觉参考
- 必须写清页面风格、字体、配色、状态和反模式
- “AI 默认紫色渐变”“大路货 SaaS 页面”这类通病要提前规避

## 6. 技能生态要用，但不能盲装

来源启发：

- `skills.sh`
- `claude-code-skills`
- `everything-claude-code`

吸收建议：

- 技能是增强层，不是主流程
- 安装前先看：
  - 解决什么问题
  - 是否安全
  - 是否真的适合当前项目
- 不把外部 skill 仓库整套搬入内部流程

## 7. 角色化 agent 适合专项协作，不适合代替主流程

来源启发：

- `agency-agents`
- `oh-my-openagent`
- `gstack`

吸收建议：

- 角色化 agent 更适合：
  - 前端专项
  - 文案专项
  - QA 复查
  - 架构评审
- `gstack` 这类流程型仓库更适合拿来定义协作节奏，不适合直接当业务底座
- 不适合直接替代 proposal/spec/design/tasks 主链路

## 三、目标项目 当前最应该定下来的内部标准

第一优先：

- `Research -> Proposal -> Spec -> Design -> Tasks -> QA -> Review -> Release -> Retro`

第二优先：

- 每个项目必须有对标和视觉参考
- 每个项目必须有验收标准
- 每个项目必须留下复盘

第三优先：

- AI 能力和 skill 体系只做增强，不替代产品和测试职责

## 四、哪些仓库最适合继续深学

### 第一梯队

- `OpenSpec`
- `superpowers`
- `web-access`
- `everything-claude-code`
- `gstack`

### 第二梯队

- `ui-ux-pro-max-skill`
- `skills.sh`
- `claude-code-skills`

### 第三梯队

- `agency-agents`
- `oh-my-openagent`

## 五、对 目标项目 的直接动作建议

1. 先把标准流程正式定稿。
2. 再把 spec / design / qa 模板补完整。
3. 选一个真实项目按这套流程完整跑一遍。
4. 跑完之后再决定哪些工具要真正引入。
