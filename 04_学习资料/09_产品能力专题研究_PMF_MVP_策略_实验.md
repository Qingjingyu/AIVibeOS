# 产品能力专题研究：PMF / MVP / 策略 / 实验

## 研究目标

这一轮研究要回答：

1. 一个产品最早期到底该先解决什么
2. PMF 应该怎么判断，而不是靠感觉
3. MVP 该怎么砍，才不会既慢又错
4. 产品策略和验证实验应该怎么进入 AIVibe OS

## 当前初步结论

基于 Lenny 知识库当前检索到的高信号内容，这条线最值得先吸收 4 个判断：

### 1. PMF 不是“产品上线了”，而是用户真的离不开

从 `how-to-know-if-youve-got-product-market-fit`、`what-it-feels-like-when-youve-found-product-market-fit` 这类内容看，PMF 的核心不是功能列表，而是：

- 用户持续回来
- 用户强烈失落感
- 需求和传播开始自然发生

### 2. MVP 不是“小一点的完整版”，而是最小验证单元

如果 MVP 不能验证关键假设，那它只是“缩水版产品”，不是好 MVP。

### 3. 产品策略不是口号，而是取舍

像 `Strategy Blocks`、`An operator’s guide to product strategy` 这类材料反复说明：

- 策略的本质是选择不做什么
- 策略要能指导资源分配
- 策略要能解释为什么先做这个，而不是那个

### 4. 实验不是增长团队专属，而是产品判断能力

从 `Fostering a culture of experimentation` 这类内容能看到，实验不是最后再补，而是：

- 用来验证关键假设
- 用来减少方向性错误
- 用来避免团队凭直觉赌大功能

## 对 AIVibe OS 的直接启发

第一，proposal 模板应更强调“问题”和“假设”，不只是“做什么功能”。  
第二，spec 模板应补“关键假设”和“验证方式”。  
第三，验收标准不应只写功能完成，还应写验证结果。  
第四，后面应单独沉淀一套产品实验模板。
