# brainstorming-and-discovery

## 解决什么问题

在正式进入 `/spec` 之前，把方向、约束、切口和 MVP 想清楚，避免一上来就写错东西。

## 什么时候用

- 需求很模糊
- 用户只给了大方向
- 方案空间明显不止一种
- 开工前需要先砍伪需求

## 不什么时候用

- 已经有清晰 Spec
- 当前是纯实现或纯修 bug 阶段

## 输入

- 用户目标
- 当前背景
- 已知约束

## 输出

- 问题定义
- 方案选项
- MVP 范围
- 下一步是 `/spec` 还是 `office-hours`

## 执行步骤

1. 先复述目标，不急着给方案。
2. 找出真正痛点和伪需求。
3. 给 2 到 3 个可行路径，并说明取舍。
4. 收敛到最小切口和成功标准。
5. 明确下一步进入 `/spec` 或升级 `office-hours`。

## 常见偷懒借口

- “先做个 demo 再说”
- “边写边想更快”
- “这个方向应该差不多没问题”

## 红线与危险信号

- 问题都没说清就想直接实现
- MVP 和愿景混在一起
- 复杂高风险问题只有一个方案选项

## 验证方式

- 能清楚写出一句话问题定义
- 有明确 MVP 和非目标
- 有推荐路径和放弃路径

## 交接产物

- discovery 摘要
- 方案对比
- 推荐方向

## 真实例子

- 新产品功能切口收敛
- 大型模糊需求开工前澄清
