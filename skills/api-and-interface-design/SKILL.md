# api-and-interface-design

## 解决什么问题

把接口、模块边界和公共契约设计成稳定、可预测、难被误用的形状。

## 什么时候用

- 新增 API 或模块接口
- 设计前后端契约
- 设计组件 props 或跨模块类型
- 修改现有公共接口

## 不什么时候用

- 纯内部一次性脚本
- 没有任何边界暴露的小型实现细节

## 输入

- 业务目标
- 调用方需求
- 现有系统约束
- 兼容性要求

## 输出

- 契约定义
- 错误语义
- 边界校验策略
- 兼容性说明

## 执行步骤

1. 先定契约，再写实现。
2. 明确输入、输出、错误语义，不混多套风格。
3. 在系统边界校验，不把验证撒到内部各层。
4. 优先新增可选字段，不轻易改旧字段语义。
5. 提前考虑未来废弃和迁移，不把实现细节暴露成默认契约。

## 常见偷懒借口

- “先把接口做出来，后面再收”
- “这只是内部接口，不用太规范”
- “先让前端能调通，错误格式以后统一”

## 红线与危险信号

- 同类接口返回不同形状
- 错误语义混乱
- 没有分页却直接暴露列表
- 直接把实现细节泄漏给调用方

## 验证方式

- 输入输出有明确结构
- 错误格式一致
- 边界校验位置清楚
- 兼容性变化有说明

## 交接产物

- 接口契约
- 错误码或错误语义说明
- 兼容性与迁移备注

## 真实例子

- REST / RPC 接口设计
- 前后端契约定义
- 模块公共方法边界设计
