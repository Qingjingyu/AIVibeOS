# spec-driven-development

## 解决什么问题

把模糊想法变成可以执行、可以验收的正式需求文档。

## 什么时候用

- 进入正式开发前
- 需要多人协作
- 任务边界容易漂移

## 不什么时候用

- 纯一次性临时探索
- 已有清晰且最新的 Spec

## 输入

- 问题定义
- 目标用户
- 约束条件
- 现有系统上下文

## 输出

- Proposal
- Spec
- 成功标准
- 边界说明

## 执行步骤

1. 先写为什么做，不先写实现细节。
2. 明确用户故事、目标、范围、非目标。
3. 写清输入输出、状态、边界和风险。
4. 明确测试和验收方式。
5. 把文档挂到标准模板和入口里。

## 常见偷懒借口

- “先实现，文档后补”
- “这个功能很小，不需要 Spec”
- “口头已经说清楚了”

## 红线与危险信号

- 没有成功标准
- 没有非目标
- 没有边界和风险说明

## 验证方式

- 能落到 [../../02_Spec模板/01_产品需求Spec模板.md](../../02_Spec模板/01_产品需求Spec模板.md)
- DoD 和 QA 入口能对上
- 实现前就能知道什么算完成
