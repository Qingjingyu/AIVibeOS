# debugging-and-error-recovery

## 解决什么问题

系统地定位问题根因，而不是靠连续试错把问题越修越乱。

## 什么时候用

- 测试失败
- 页面或接口行为异常
- 构建、发布或集成出错

## 不什么时候用

- 需求还没进入实现
- 当前没有具体故障，只是在做预防性设计

## 输入

- 错误现象
- 复现步骤
- 相关日志或报错

## 输出

- 根因判断
- 最小修复方案
- 防复发建议

## 执行步骤

1. 先复现，不复现就继续收证据。
2. 看报错、日志、最近改动，不先改代码。
3. 缩小问题范围，判断是数据、状态、环境还是逻辑。
4. 只做最小修复，不同时改很多处。
5. 修完补验证和防护。

## 常见偷懒借口

- “先试着改一下看看”
- “这个报错大概就是那个原因”
- “一口气把相关地方全改了”

## 红线与危险信号

- 还没定位根因就开始修
- 同一个问题连续多次盲改
- 用更多复杂度掩盖真正问题

## 验证方式

- 有清楚的复现和修复前后差异
- 修复对应根因，不只是掩盖症状
- 相关测试或运行验证通过

## 交接产物

- 复现步骤
- 根因说明
- 修复与验证结果

## 真实例子

- 接口异常排查
- 页面状态错乱修复
- 构建失败定位
