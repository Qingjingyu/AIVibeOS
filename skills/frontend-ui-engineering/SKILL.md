# frontend-ui-engineering

## 解决什么问题

把用户可见界面做成生产级 UI，而不是“AI 味很重、勉强能看”的页面。

## 什么时候用

- 新页面开发
- 新组件开发
- 现有 UI 改版或质量修复
- 响应式与交互状态实现

## 不什么时候用

- 完全无 UI 的任务
- 只做数据层或纯文档任务

## 输入

- 设计稿或页面目标
- 现有设计系统
- 真实文案和状态需求

## 输出

- 可运行 UI
- 可访问性和响应式处理
- 浏览器可见证据

## 执行步骤

1. 先确认设计系统和页面层级，不先堆视觉花活。
2. 组件尽量小而清楚，数据和展示分开。
3. 用真实内容和真实状态，不靠假文案掩盖问题。
4. 处理好键盘可达、空状态、错误状态、加载状态。
5. 做完后在浏览器里看，而不是只看 JSX / CSS。

## 常见偷懒借口

- “先做个差不多的页面”
- “后面再补响应式和可访问性”
- “先用默认渐变和圆角撑一下效果”

## 红线与危险信号

- 页面有明显 AI 套路感
- 只有默认 happy path，没有空状态和错误状态
- 交互元素不可键盘访问
- 只用颜色表达信息

## 验证方式

- 页面在浏览器里可见且可操作
- 移动和桌面至少基本可用
- 关键状态都有表现
- 设计系统或视觉语言一致

## 交接产物

- 页面或组件截图
- 状态说明
- 已知差异或限制

## 真实例子

- 产品页面实现
- 管理后台组件实现
- Figma 到页面的 UI 落地
