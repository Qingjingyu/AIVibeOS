# code-simplification

## 解决什么问题

在不改变精确行为的前提下，降低复杂度、减少特殊分支和维护成本。

## 什么时候用

- 代码已经能工作，但太绕
- review 发现复杂度过高
- 重复逻辑开始增多

## 不什么时候用

- 行为还不稳定
- 需求还在频繁变

## 输入

- 当前实现
- 现有行为证据
- 相关测试

## 输出

- 行为不变但更简单的实现
- 简化前后说明

## 执行步骤

1. 先确认现有行为，不先乱删。
2. 找重复、特例、深层分支。
3. 用更通用的结构替代特殊情况。
4. 保留验证，确保行为不漂。
5. 明确说明简化了什么、没改什么。

## 常见偷懒借口

- “这块虽然丑，但先别碰”
- “重写更快”
- “简化会不会影响现有功能，先算了”

## 红线与危险信号

- 没有行为证据就重写
- 简化后无法证明没变
- 为了“好看”引入更大改动

## 验证方式

- 相关测试通过
- 复杂度确实下降
- 行为差异为零或有明确说明

## 交接产物

- 简化前后摘要
- 风险说明
- 验证结果

## 真实例子

- 深层条件分支收敛
- 重复组件逻辑提纯
