# documentation-and-adrs

## 解决什么问题

把“为什么这么做”和“别人接手时需要知道什么”记下来，避免系统只剩代码和口头记忆。

## 什么时候用

- 做了结构性决策
- 引入新约束或新边界
- 准备交接
- 发布后需要沉淀结论

## 不什么时候用

- 临时试验且不准备沉淀
- 当前没有任何值得长期保留的决策

## 输入

- 当前改动或决策
- 影响范围
- 需要被谁理解

## 输出

- 文档更新
- ADR 或决策记录
- 交接说明

## 执行步骤

1. 先判断这次改动是不是会影响后续理解。
2. 如果会，优先补“为什么”，不只补“做了什么”。
3. 更新最近的入口和索引，不新造孤立文档。
4. 把长期有效的结论和短期任务区分开。
5. 确保别人脱离当前对话也能接住。

## 常见偷懒借口

- “代码里已经能看出来”
- “后面统一整理文档”
- “这次先不记，大家都知道”

## 红线与危险信号

- 关键决策只存在于聊天里
- 改了流程但没改入口
- 交接依赖口头补充

## 验证方式

- 最近入口已更新
- 关键决策有记录
- README / 索引 / 交接路径可用

## 交接产物

- 更新后的入口
- 决策说明
- 后续维护提示

## 真实例子

- 架构决策记录
- 系统规则更新后同步索引
