# git-workflow-and-versioning

## 解决什么问题

让改动能被安全保存、清楚审查、方便回退，而不是做成一坨不可读的历史。

## 什么时候用

- 准备提交
- 准备开 PR
- 需要整理改动边界

## 不什么时候用

- 纯只读分析
- 当前环境明确不要求 git 交付

## 输入

- 当前改动
- 提交意图
- 关联任务范围

## 输出

- 清晰的提交边界
- 合理的提交信息
- 便于 review 的改动历史

## 执行步骤

1. 先确认本轮到底改了什么。
2. 不把无关改动混进同一提交。
3. 提交信息要能说明行为变化。
4. 尽量保持小而清楚的提交。
5. 任何危险 git 动作先停下来判断。

## 常见偷懒借口

- “先全部一起提交”
- “提交信息随便写一下”
- “历史后面再整理”

## 红线与危险信号

- 一个提交塞太多不相关内容
- 提交信息看不出改了什么
- 用破坏性 git 动作掩盖问题

## 验证方式

- 提交边界清楚
- 提交信息能独立理解
- review 时能按提交读懂变化

## 交接产物

- 提交摘要
- 关联任务
- 需要关注的历史点
