# idea-refine

## 解决什么问题

把模糊想法收敛成可写 Proposal / Spec 的明确问题。

## 什么时候用

- 需求还很模糊
- 只知道方向，不知道切口
- 用户目标和成功标准还不清楚

## 不什么时候用

- 已经有清晰 Spec
- 现在是纯实现阶段

## 输入

- 用户目标
- 当前背景
- 约束条件

## 输出

- 问题定义
- MVP 范围
- 成功标准
- 下一步 Proposal / Spec 输入

## 执行步骤

1. 先复述目标，不急着给方案。
2. 识别真实痛点、伪需求、边界和约束。
3. 区分必须做和可以后做。
4. 给出一句话问题定义和最小切口。
5. 明确下一步是进入 Proposal 还是 Spec。

## 常见偷懒借口

- “先做出来再看”
- “边写边想”
- “反正后面还能改”

## 红线与危险信号

- 连用户要解决的问题都说不清
- MVP 和 V2 混在一起
- 目标和验证标准没有分开

## 验证方式

- 能写出一句话问题定义
- 能列出 Must-have 和 Nice-to-have
- 能明确下一步文档入口

## 交接产物

- 一段问题定义
- 一组约束
- 一份 MVP 范围说明

## 真实例子

- 新产品立项前的需求收敛
- 复杂功能开始前的方向澄清
