# planning-and-task-breakdown

## 解决什么问题

把 Spec 拆成小而可验证的任务，避免“大任务一把梭”。

## 什么时候用

- Spec 已明确
- 任务要交给多人或多代理
- 需要控制风险和节奏

## 不什么时候用

- 问题还没定义清楚
- 只是 5 分钟级别的小修补

## 输入

- Spec
- 技术约束
- 现有代码上下文

## 输出

- 任务列表
- 依赖顺序
- 每步验收标准

## 执行步骤

1. 先按垂直切片拆，不先按技术层拆。
2. 每个任务只做一件能验收的事。
3. 标出依赖顺序和并行可能。
4. 给每步补最小验证方式。
5. 明确哪些任务必须本地做，哪些可后台 / 云端。

## 常见偷懒借口

- “先改一大块再整理”
- “这些步骤拆太细太麻烦”
- “测试最后一起跑”

## 红线与危险信号

- 一步里同时改很多层
- 没有验收标准
- 没有依赖顺序

## 验证方式

- 任务能落到 [../../02_Spec模板/05_Tasks模板.md](../../02_Spec模板/05_Tasks模板.md)
- 每步都有输入、输出、验证
- 能分出本地 / 后台 / 云端模式
