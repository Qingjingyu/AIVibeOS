# source-driven-development

## 解决什么问题

让实现和技术决策尽量基于一手来源，而不是靠记忆、猜测或二手博客。

## 什么时候用

- 接入新框架或新能力
- API、SDK、CLI 用法不确定
- 版本差异可能影响实现

## 不什么时候用

- 当前任务完全是仓内纯文档整理
- 所需事实已经在本仓有明确结论且没有时效风险

## 输入

- 要验证的问题
- 框架 / 工具名称
- 当前版本或目标版本

## 输出

- 一手来源链接
- 已验证结论
- 未确认项说明

## 执行步骤

1. 先明确要验证的具体问题，不宽泛搜索。
2. 优先看官方文档、官方仓库、源码或官方 cookbook。
3. 把“已确认”和“推断”分开写。
4. 只把对当前任务有直接影响的结论回写到实现或文档。
5. 如果资料冲突，明确记录冲突点，不私自拍板。

## 常见偷懒借口

- “我记得是这样”
- “网上差不多都这么写”
- “先按以前经验做，错了再改”

## 红线与危险信号

- 高时效问题不查一手来源
- 把二手文章当最终依据
- 不区分已验证和未验证内容

## 验证方式

- 至少有一个一手来源
- 关键结论能对应到官方文档或源码
- 不确定项被明确标记

## 交接产物

- 关键来源链接
- 结论摘要
- 风险和未验证项

## 真实例子

- 新增 SDK 接入前查官方用法
- 新的 CLI / MCP 配置前查官方文档
