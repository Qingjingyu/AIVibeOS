# deprecation-and-migration

## 解决什么问题

让旧系统、旧接口、旧实现的下线和迁移有计划地发生，而不是长期变成僵尸代码。

## 什么时候用

- 旧实现要被新实现替代
- 旧 API / 旧模块要废弃
- 有重复系统准备合并
- 老代码已经没人维护但仍被依赖

## 不什么时候用

- 还没有替代方案
- 当前只是小范围重构且没有对外影响

## 输入

- 被废弃对象
- 当前使用者范围
- 替代方案
- 迁移成本

## 输出

- 废弃决策
- 迁移计划
- 迁移说明
- 下线条件

## 执行步骤

1. 先判断旧东西是否还有独特价值。
2. 没有替代方案时，不宣布废弃。
3. 量化有多少消费者、迁移难度多高。
4. 先让替代方案在真实环境可用，再逐步迁移。
5. 只有确认零活跃依赖后，才真正删除旧系统和相关文档。

## 常见偷懒借口

- “它还能跑，先留着”
- “先发废弃通知，用户自己会迁”
- “两个版本先长期共存”

## 红线与危险信号

- 废弃没有迁移指南
- 没有使用量证据就删旧系统
- 旧系统已经没人维护但继续放着
- 新功能还加在废弃系统上

## 验证方式

- 替代方案已可用
- 有明确迁移步骤
- 活跃使用量被确认
- 删除前后有证据

## 交接产物

- 废弃通知
- 迁移指南
- 删除条件和完成记录

## 真实例子

- 旧接口迁到新接口
- 旧模块被新模块替换
- 长期无人维护代码正式下线
