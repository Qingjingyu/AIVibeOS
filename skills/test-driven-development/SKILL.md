# test-driven-development

## 解决什么问题

用测试证明行为正确，而不是靠主观感觉说“应该没问题”。

## 什么时候用

- 新功能开发
- Bug 修复
- 高风险重构

## 不什么时候用

- 纯探索性草图
- 文档类任务

## 输入

- 需求或 Bug 描述
- 预期行为
- 现有测试环境

## 输出

- 失败测试
- 最小实现
- 通过测试

## 执行步骤

1. 先写能暴露问题的测试。
2. 运行它，确认真的失败。
3. 写最小实现让测试通过。
4. 重新跑相关测试。
5. 必要时再做小幅重构。

## 常见偷懒借口

- “这个太简单，不用测”
- “先实现，后面再补”
- “我肉眼看过了”

## 红线与危险信号

- 没有办法证明修复前后差异
- 修复后不跑测试
- 用大范围手测替代关键自动化验证

## 验证方式

- 有失败到通过的证据
- 测试名字能说明行为
- 相关用例都跑过

## 交接产物

- 新增或更新的测试
- 测试命令
- 测试结果

## 真实例子

- Bug fix
- 表单逻辑和接口行为验证
