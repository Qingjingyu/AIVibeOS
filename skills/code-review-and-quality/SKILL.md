# code-review-and-quality

## 解决什么问题

在合并前发现行为风险、结构问题和质量退化，而不是等上线后补锅。

## 什么时候用

- 代码准备合并
- 大改动完成后
- 多代理输出需要收口

## 不什么时候用

- 需求还没定
- 改动还在半成品状态

## 输入

- 改动内容
- 测试结果
- 任务背景

## 输出

- findings
- 风险等级
- 是否可合并判断

## 执行步骤

1. 先看行为有没有变坏。
2. 再看边界、异常、数据流和兼容性。
3. 再看结构、复杂度和重复。
4. 只提出有依据的问题，不表演式 review。
5. 结论必须基于证据，不靠感觉。

## 常见偷懒借口

- “能跑就先合”
- “这只是内部代码”
- “以后再 review”

## 红线与危险信号

- 没有测试就想合并
- 高风险逻辑没有说明
- 明显复杂度上升却没人处理

## 验证方式

- findings 带文件和原因
- 风险分级清楚
- 能区分必须修和可后续优化

## 交接产物

- review 结论
- findings 列表
- 未解决风险

## 真实例子

- 合并前代码审查
- 多代理产出统一收口
