# Splash Figma 到代码完整案例

## 这个案例的定位

这份案例不是抽象原则，也不是一句“照着 Figma 做”。

它的目标是把一次真实复杂页面的高还原过程完整摊开，让别人能直接照着走：

- 输入是什么
- 为什么判断成复杂页
- 为什么必须先拆层
- 参数该怎么拿
- 每一轮怎么推进
- 最后如何验收

## 一、案例输入

- Figma 文件：`Sonora - Audio Book`
- file key：`enBbH6DF4uLDWpDp3b5yhy`
- 完整态节点：`5162:28785`
- 本地路由：`/splash`
- 本地页面：`http://localhost:3000/splash`

## 二、为什么这页必须走复杂页流程

这页同时具备：

- 大面积半透明壳体
- 白雾 / 玻璃感
- 多张卡片穿插
- 光效层
- 装饰漂浮物
- 多个视觉重心

只要一个页面同时出现这些特征，就不要直接写代码。

第一判断就应该是：

**这页是复杂页，必须先拆层。**

## 三、这次真正用到的能力

### 1. Figma MCP

作用：

- 拉完整截图
- 拉分层截图
- 拉设计参数
- 必要时在 Figma 里建工作台

### 2. 浏览器侧验收

作用：

- 打开本地页面
- 看真实渲染效果
- 固定 viewport 截图
- 对稿

### 3. 本地截图链路

作用：

- 留下每一轮证据
- 方便复盘
- 防止“感觉像了”

## 四、这次在 Figma 里怎么拆层

真正还原时，重点拆成 5 层：

1. `Background`
2. `Glow`
3. `Decor`
4. `Cover Stack`
5. `Brand Core`

这样拆的价值很直接：

- 设计层次说清楚了
- 代码结构可以一一对应
- 每一轮只需要解决一个主要问题

## 五、开工前应该先拿什么参数

这次最关键的经验之一，是复杂页不能只拿截图。

至少要对关键层整理出参数表，优先拿：

- `x / y / width / height`
- `radius`
- `opacity`
- `blur`
- `shadow`
- `gradient`
- `z-index`
- `mask 边界`
- `backdrop blur`
- `原始素材路径`
- `图片 crop 方式`

如果这些不先拿，最后最常见的情况就是：

- 位置看着差不多
- 质感始终不对
- 一直卡在“好像就差一点”

## 六、代码怎么映射分层

代码层直接按拆层结果建立：

- `splash-background`
- `splash-glow`
- `splash-decor`
- `splash-covers`
- `splash-brand`

这样一来，一旦发现“品牌核不对”，就知道该改哪层，不会全页一起乱动。

## 七、每一轮怎么推进

### 第一轮：先起骨架

- 页面先能跑
- 主要层先摆出来
- 不追求一次到位

### 第二轮：收最伤观感的问题

优先改：

- 视觉重心
- 卡片露出比例
- 光效范围
- 主要阴影关系

### 第三轮：复杂层单独收差

只盯最关键的一层，例如：

- 品牌核
- 卡片吞入关系
- 白雾遮罩

不要一轮里全页乱改。

## 八、这个案例真正教会什么

这份案例最值钱的，不是“把 Splash 做出来”。

而是下面这条方法：

**复杂页面应该先拆层、再拿参数、再分轮收差。**

如果你把这条学会了，后面碰到别的复杂页，也不会再从零摸索。
