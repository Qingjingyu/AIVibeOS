# Figma MCP + 浏览器真值闭环

## 1. 这份文档解决什么

这份不是基础 SOP 的重复版。

它只解决一个升级问题：

**为什么高保真还原不能只看 Figma，还必须看浏览器。**

也就是说：

- `Figma MCP` 负责回答“设计稿到底是什么”
- 浏览器验收负责回答“代码现在到底成什么样了”

这两边缺一边，都会翻车。

## 2. 两种真值分别负责什么

### Figma 真值负责

- 页面截图
- 结构层级
- 节点尺寸
- 素材入口
- 参数真值

典型能力：

- `get_design_context`
- `get_screenshot`
- `get_metadata`
- 必要时 `use_figma`

### 浏览器真值负责

- 页面真实渲染结果
- 字体和模糊在浏览器里的观感
- 层级是否穿帮
- 交互是否真能点
- 当前 viewport 下是不是已经跑偏

可以通过：

- 浏览器自动化
- 本地截图脚本
- 手动真实页面检查

## 3. 标准闭环

推荐闭环是：

`Figma 截图 -> 设计上下文 -> 页面实现 -> 浏览器截图 -> 差异记录 -> 下一轮修正`

复杂页变成：

`Figma 截图 -> 分层截图 -> 关键层参数 -> 页面分层实现 -> 浏览器截图 -> 差异记录 -> 下一轮修正`

## 4. 什么时候必须升级成闭环模式

下面这些场景，不能只用基础 SOP，必须明确走闭环：

1. 页面本身追求高保真
2. 页面有复杂视觉材料
3. 页面已经做过一版，但“总觉得味道不对”
4. 页面改了 2 轮以上，还在靠感觉修

## 5. 浏览器真值为什么这么重要

很多人以为拿到 Figma 参数就够了。

实际上最容易失真的地方，经常都发生在浏览器侧：

- 模糊发脏
- 透明壳体发灰
- 卡片像浮在外面
- 阴影过重
- 字体节奏变硬

这些问题只看 Figma 说明文档是发现不了的。
必须看真实浏览器截图。

## 6. 闭环怎么跑

### 第一步：拿设计真值

先拿：

- 全页截图
- 关键层截图
- design context

### 第二步：先实现，不急着宣布完成

把页面先做成：

- 结构正确
- 素材正确
- 视觉方向正确

### 第三步：截浏览器真图

固定 viewport：

- 打开本地页面
- 截图
- 留图

### 第四步：只找最明显差异

一轮只抓最关键的 3 到 5 个问题，例如：

- 视觉重心偏了
- 卡片露出比例错了
- 雾化层太假
- 底部节奏太挤

### 第五步：再修一轮

不要十几处一起乱改。

## 7. 这份闭环最重要的判断

如果页面越改越像 demo，而不是越改越像原稿，通常不是你参数调得不够多，而是你漏了下面其中一项：

1. 没拆层
2. 没拿关键层参数
3. 没看浏览器截图
4. 一轮里改太多

## 8. 工具角色定义

### Figma MCP

负责：

- 设计真值
- 结构真值
- 素材真值

### 浏览器侧工具

负责：

- 当前渲染真值
- 当前差异真值
- 当前交互真值

## 9. 一句话总纲

**Figma MCP 负责看设计稿，浏览器验收负责看结果；高保真不是单边判断，是双边闭环。**
