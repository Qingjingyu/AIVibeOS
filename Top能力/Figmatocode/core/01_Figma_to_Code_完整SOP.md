# Figma to Code 完整 SOP

## 1. 目标

这份 SOP 解决的是“如何稳定把一个 Figma 页面落成代码”，不是“如何让 AI 随机生成一个看起来像 UI 的东西”。

目标只有三个：

1. 输入要准
2. 过程要稳
3. 结果要能证明

## 2. 先说结论

标准主线是：

`Figma 页面 -> 原始上下文 -> 原始素材 -> 页面分型 -> 结构拆层 -> 页面实现 -> 浏览器截图对稿 -> 工程验证`

复杂页增加一道：

`关键层参数表 -> 分层实现`

翻成人话：

- 先把设计稿真正是什么搞清楚
- 再决定页面该怎么做
- 最后用浏览器证据证明它现在成了什么样

## 3. 不要做什么

下面这些做法默认算错：

1. 只看截图就直接开写
2. 随手换图标、换封面、换文案
3. 用整页截图糊背景交差
4. 结构还没对，就开始狂调 blur 和阴影
5. 没截图对稿，就说“已经像素级”

## 4. 必须拿到的输入

### 必要输入

- Figma URL
- file key
- node id
- 目标页面路由

### 强烈建议同时拿到

- `design context`
- `screenshot`
- `metadata`
- 原始图片
- 原始 SVG
- 变量 / token

### 缺关键输入时怎么处理

1. 缺 `screenshot`
   不进入高保真验收。
2. 缺 `design context`
   先只做结构版，不做细节精修。
3. 缺原始素材
   先补素材，再继续。

## 5. 页面先分型

不是所有页面都一个做法。

### A 类：主视觉页

适合：

- Splash
- 欢迎页
- 品牌页

重点：

- 最怕不像
- 主视觉优先

### B 类：混合交互页

适合：

- 播放页
- 分享页
- 结果承接页

重点：

- 外观要像
- 交互要真

### C 类：强业务页

适合：

- 创作页
- 列表页
- 设置页

重点：

- 结构和状态优先
- 复杂装饰局部素材化

## 6. 标准执行顺序

### Step 0：确认任务是“还原”

如果任务是还原：

- 默认不改设计
- 默认不改文案
- 默认不改素材

### Step 1：拉设计上下文

至少做：

1. 拉 `design context`
2. 拉 `screenshot`
3. 拉 `metadata`

目的：

- 看页面块
- 看层级关系
- 看哪些是图片，哪些是样式

### Step 2：下载原始素材

原始素材优先级：

1. 原始 PNG / JPG
2. 原始 SVG
3. 原始图标

只有拿不到时才允许降级。

### Step 3：先拆结构

每页至少先拆成：

- 背景层
- 内容层
- 覆盖层
- 状态层

复杂页再往下拆子层。

### Step 4：冻结状态

至少写清楚：

- 初始态
- 加载态
- 成功态
- 失败态
- 空态

如果本轮只做静态页，也要明确写出来。

### Step 5：先搭结构，再接素材，再调参数

顺序固定：

1. 搭骨架
2. 放模块
3. 接素材
4. 收参数

### Step 6：优先收关键视觉参数

先收：

- 宽高
- 间距
- 圆角
- 色值
- 渐变
- 阴影
- 模糊
- 字号 / 字重 / 行高

### Step 7：做浏览器截图对稿

必须做：

- 固定 viewport
- 截当前页面
- 和 Figma 截图对比
- 写出最明显差异

### Step 8：工程验证

至少确认：

- 页面可运行
- 构建通过
- 基本测试通过

## 7. 什么时候要走复杂页工作流

只要页面里出现下面任一情况，就要警惕：

- 毛玻璃
- 白雾
- 蒙版
- 多层卡片穿插
- 半透明壳体
- 强光效
- 多视觉重心

这时不要直接硬写。
先看 `core/03_复杂页关键层参数工作流.md`。

## 8. 完成标准

完成不等于“代码写出来了”。

默认完成标准至少包括：

1. 用了正确素材
2. 结构没有跑偏
3. 关键视觉方向正确
4. 页面可运行
5. 有浏览器截图证据
6. 还有差异时明确写差异

## 9. 一句话总纲

**先拿标准答案，再写代码；先看真实浏览器，再宣布完成。**
