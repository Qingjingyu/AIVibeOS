# 复杂页关键层参数工作流

## 1. 为什么这份文档单独存在

普通页面靠基础 SOP 就够了。

复杂页不一样。

复杂页最常见的死法不是“不会写页面”，而是：

- 背景像了，壳体不对
- 壳体像了，白雾不对
- 白雾像了，卡片吞入关系不对
- 参数全靠猜，结果越修越脏

所以复杂页必须单独加一道流程：

**先把关键层参数拿准，再开始实现。**

## 2. 什么算复杂页

只要页面出现下面任一情况，就先按复杂页处理：

- 毛玻璃
- 蒙版
- 白雾
- 大面积模糊
- 多层卡片穿插
- 半透明壳体
- 强光效
- 复杂图片裁切

## 3. 复杂页的正确顺序

不是：

`拿截图 -> 直接开写 -> 写歪了再硬调 blur`

而是：

`拿全页截图 -> 拆关键层 -> 拿关键层参数 -> 选实现方式 -> 分层实现 -> 浏览器对稿`

## 4. 先拆关键层

先把这次真正影响观感的层列出来。

常见例子：

- 背景层
- 光效层
- 装饰层
- 卡片层
- 品牌核
- 蒙版层
- 弹层

如果关键层都没拆清，不要碰样式细节。

## 5. 关键层至少要拿哪些参数

每层至少确认：

- `x / y / width / height`
- 圆角
- 透明度
- 颜色 / 渐变
- 阴影
- 模糊
- 混合模式
- 蒙版信息
- 图片裁切方式
- 原始素材路径

真正最容易失真的，一般是：

1. `opacity`
2. `blur`
3. `shadow`
4. `gradient`
5. `mask 边界`
6. `crop 方式`
7. `z-index`
8. `backdrop blur`

## 6. 每层先判断实现方式

不要一上来就默认纯 CSS。

每层都先判断更适合：

- 纯 CSS
- HTML + CSS
- 原始 PNG / JPG
- 原始 SVG
- CSS + 原始素材混合
- 单独子组件

### 一条铁规则

**复杂页不是“所有东西都代码化”才高级，而是“每层选最不容易翻车的实现方式”。**

## 7. 参数表为什么不是可选项

复杂页如果没有参数表，最常见的问题是：

- 这轮你觉得错在 blur
- 下一轮别人觉得错在 shadow
- 再下一轮又怀疑是裁切

结果三轮下来，谁都说不清到底哪层有问题。

参数表的价值不是增加格式负担，而是把“猜”变成“证据”。

## 8. 什么时候必须停下来补参数表

出现下面任一情况，立刻停：

1. 同类问题修了 2 轮还明显不像
2. 已经开始靠加更多渐变和阴影补洞
3. 自己都说不清错的是哪一层

这时不要继续调代码。
先回去补关键层参数。

## 9. 配套文件

实际执行时，直接配合：

- `templates/关键层参数表模板.md`
- `examples/Splash_Figma到代码完整案例.md`

## 10. 一句话总纲

**复杂页不是先写代码再修参数，而是先把关键层参数拿准，再分层落代码。**
