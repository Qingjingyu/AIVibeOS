# DESIGN.md B 端产品型示例

这个示例适合：

- SaaS 控制台
- 管理后台
- 工作台
- 数据面板
- 运营系统

目标不是“看起来很炫”，而是：

- 信息清楚
- 操作稳定
- 主次分明
- 能长时间使用不累
- 页面一多也不乱

---

```md
# DESIGN.md

## 1. 产品气质

### 目标感觉
- 这是一个专业、稳定、可信、效率优先的 B 端产品。
- 用户打开页面后，应该立刻感受到：信息可控、操作清楚、系统靠谱。
- 整体更像成熟工作台，不像营销页，也不像实验性设计作品。

### 不要变成什么
- 不要像视觉练习作品。
- 不要像信息很多但层级很乱的老式后台。
- 不要为了所谓高级感，牺牲操作效率和扫读速度。

## 2. 色彩与 Token

### 主色策略
- 主色只用于关键交互、焦点态、当前选中和少量高优先级信息。
- 不要让主色在表格、列表、侧栏里到处泛滥。
- 主色的任务是“提示重点”，不是“染满系统”。

### 中性色与背景
- 页面主背景以安静中性色为主。
- 卡片、表格、筛选区之间要靠轻层次区分，而不是重装饰。
- 边框可以用，但要克制，目标是帮助分区，不是做视觉噪音。

### 语义色
- 成功、警告、错误、信息提示必须有明确区分。
- 风险操作区域可以加强对比，但不要做得像报警灯。

## 3. 字体与层级

### 排版原则
- 标题层级清楚，但整体偏克制。
- 正文、表格、表单说明必须优先可读性。
- 数字、金额、比例、状态要比装饰更重要。

### 层级建议
- 页面标题：说明当前模块是什么
- 区块标题：说明当前区域要做什么
- 正文和说明：帮助用户快速理解，不写长篇大论
- Caption：用于次要提示、更新时间、字段说明

### 禁忌
- 不要一会儿超大标题，一会儿超小正文，层级失控。
- 不要用太多弱灰文字导致用户看不清。

## 4. 关键组件风格

### Button
- 主按钮要明确、稳定、易识别。
- 次按钮和文字按钮要区分清楚，避免用户误点关键操作。
- 危险按钮必须和普通按钮有明显区别。

### Form / Filter
- 表单和筛选区优先追求清楚、紧凑、可扫读。
- Label、placeholder、默认值、错误提示要各司其职，不能混成一团。
- 高级筛选可以折叠，但默认常用项要直接可见。

### Table / List
- 表格是核心阅读区，要优先保证对齐、密度、状态展示和操作列稳定。
- 列表项和表格行 hover 要清楚，但不要过重。
- 批量操作、排序、筛选、分页等操作要像一个体系，不要每页都不一样。

### Card / Stat / Panel
- 卡片用于容纳摘要信息、分区信息、图表模块。
- 统计卡片重点是数字和趋势，不要让装饰盖过数据。
- 面板式区域要稳定，适合长时间使用。

### Modal / Drawer
- 用于补充配置、确认操作、轻量编辑。
- 复杂流程优先新页面，不要过度堆弹层。

### Sidebar / Topbar
- 侧栏要清楚表达层级和当前位置。
- 顶栏只放高频全局动作，不要塞一堆不常用入口。

## 5. 布局与留白

### 页面结构
- B 端页面默认以“导航区 + 内容区 + 操作区”组织。
- 内容区优先按任务组织，而不是按好看组织。
- 一个页面只保留一个主任务，其他内容作为辅助。

### 留白策略
- 不求大留白，但求节奏稳定。
- 高密度页面也必须保留最基本的呼吸感，避免压迫。
- 移动端优先保留查询、主要信息、核心操作，次级面板可以折叠或下沉。

## 6. 状态与动效

### 必须覆盖的状态
- Loading：列表、表格、图表、详情页都要有明确加载态
- Empty：说明为什么空，以及下一步能做什么
- Error：说明出了什么问题、是否可重试、是否影响数据正确性
- Disabled：说明为什么不可用
- Success / Warning：用于反馈操作结果和系统风险

### 动效原则
- 动效只用于帮助理解状态变化，例如展开、切换、刷新、保存反馈。
- 节奏要快、短、稳。
- 不要使用大幅位移、弹跳、飘浮等会打断操作的动效。

## 7. Do / Don't

### Do
- Do: 先保证信息结构清楚，再谈风格
- Do: 保持组件行为一致，避免同类组件多套规则
- Do: 默认补齐加载、空态、报错态、无权限态
- Do: 优先让高频任务更短、更稳、更少思考

### Don't
- Don't: 为了显得现代就乱用玻璃拟态、重阴影、大渐变
- Don't: 把重要信息藏进二级交互
- Don't: 一个系统里做出多套按钮、表格、筛选风格
- Don't: 只做正常路径，不做异常反馈

## 8. 给 Agent 的执行提示

- 如果没有明确视觉参考，优先生成成熟 SaaS 工作台风格，不要擅自做艺术化表达。
- 页面中最该突出的通常是：当前任务、关键数据、关键操作。
- 所有新页面都优先复用已有表单、表格、卡片、筛选模式。
- 如果设计选择会影响扫读效率、操作稳定性或状态清晰度，优先牺牲花样，不牺牲效率。
```

---

## 一句话原则

**B 端设计的第一职责不是漂亮，而是让人稳定、高效、低负担地完成工作。**
