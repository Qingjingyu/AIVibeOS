# 如何写一个好 Spec

## 目的

这份文档不是模板，而是教团队怎么把 spec 写到“AI 和人都不用猜”的程度。

## 一、什么叫好 Spec

好 spec 不是字多，而是：

- 边界清楚
- 状态完整
- 文案明确
- 验收可执行
- AI 拿到后不会自由发挥

## 二、写 Spec 前先做什么

必须先完成：

- `Research`
- `Proposal`

没有 research，spec 容易拍脑袋。  
没有 proposal，spec 容易失控。

## 三、Spec 必须写清楚的 8 件事

### 1. 页面清单

必须列出到底做哪些页面，不要只写“做一个功能”。

### 2. 用户主流程

必须写：

- 用户从哪里进入
- 用户点什么
- 系统返回什么
- 成功后去哪里
- 失败时怎么办

### 3. 状态清单

每个页面至少要考虑：

- 默认状态
- loading 状态
- 空状态
- 错误状态
- 成功状态
- 禁用状态

很多 demo 感，都是因为状态没写。

### 4. 文案

不能写“这里给个提示”，要直接写：

- 按钮文案
- 错误文案
- Toast 文案
- 空状态文案

### 5. 交互细节

必须明确：

- 点击后是否防重
- 是否弹窗确认
- 是否自动跳转
- 是否允许返回
- 是否允许重试

### 6. 接口口径

至少写：

- 请求方式
- 请求参数
- 返回结构
- 失败场景
- 超时处理

### 7. 视觉要求

必须明确：

- 风格参考
- 字体方向
- 配色方向
- 页面气质
- 明确禁止的 demo 风格

### 8. 验收标准

没有验收标准的 spec，不算完整 spec。

至少要写：

- 功能验收
- 交互验收
- 视觉验收
- 异常验收

## 四、最常见的坏 Spec

### 坏 Spec 1

“做一个跟 XX 差不多的页面”

问题：

- 边界不清
- 交互不清
- 文案不清
- AI 只能猜

### 坏 Spec 2

“有错误时提示用户”

问题：

- 提示什么
- 用什么形式提示
- 要不要可重试
- 错误后停留在哪

全都没写。

## 五、Eoto AI 内部要求

以后写 spec，至少满足下面三条：

1. 不允许只有目标，没有页面与状态
2. 不允许只有功能，没有文案与交互
3. 不允许只有流程，没有验收标准

## 六、一句话标准

**好 spec 的标准不是“写完了”，而是“别人和 AI 拿到后不用猜”。**
