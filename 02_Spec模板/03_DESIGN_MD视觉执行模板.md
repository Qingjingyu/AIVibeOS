# DESIGN.md 视觉执行模板

## 适用场景

这个模板用于项目根目录下的 `/DESIGN.md`。

适合这些情况：

- 没有完整 Figma，但需要统一页面风格
- 有 Figma，但只覆盖核心页，其他页面需要统一视觉边界
- 要做官网、营销页、品牌页、活动页或早期原型
- 多个 Agent 会并行生成页面，需要统一审美约束

它不能替代：

- `Proposal`
- `Spec`
- 技术 `Design`
- 高保真 Figma

一句话：

**它负责定义“页面应该长什么感觉”，不负责定义“功能怎么做”。**

---

```md
# DESIGN.md

## 1. 产品气质

### 目标感觉
- 这个产品应该给人的第一感觉是：[专业 / 可信 / 轻盈 / 高级 / 温暖 / 冷静 / 效率型]
- 它更像：[专业工具 / 品牌官网 / 创作者产品 / 数据平台 / 消费应用]
- 整体优先追求：[清晰 / 稳定 / 情绪感 / 转化 / 品牌辨识度]

### 不要变成什么
- 不要像：[廉价营销页 / 黑客马拉松 demo / 过度娱乐化产品 / 传统企业后台]
- 不要为了“高级感”牺牲：[可读性 / 信息密度 / 反馈清晰度]

## 2. 色彩与 Token

### 主色策略
- 主色：[`token名` 或颜色用途说明]
- 主色只用于：[关键按钮 / 高亮数据 / 链接 / 焦点态]
- 不要把主色用于：[整页大面积铺底 / 所有按钮 / 所有图标]

### 中性色与背景
- 页面背景以：[浅色中性 / 深色中性 / 双色背景] 为主
- 卡片背景：[纯白 / 浅灰 / 半透明 / 深色面板]
- 边框颜色倾向：[弱边框 / 明确边框 / 几乎不用边框]

### 语义色
- 成功态：[说明]
- 警告态：[说明]
- 错误态：[说明]
- 语义色必须做到一眼能区分，不要只靠文字补救

## 3. 字体与层级

### 排版原则
- 标题风格：[克制 / 强对比 / 品牌感明显]
- 正文风格：[高可读 / 紧凑 / 轻盈]
- 数字与数据展示：[强调精确 / 强调冲击 / 强调稳定]

### 层级建议
- H1 用于：[页面主标题 / Hero 标题]
- H2/H3 用于：[区块标题 / 模块标题]
- Body 用于：[正文 / 说明 / 列表]
- Caption/Hint 用于：[辅助文案 / 表单说明 / 次要信息]

### 禁忌
- 不要同时混用太多字体风格
- 不要出现标题很花但正文难读的情况

## 4. 关键组件风格

### Button
- 主按钮应该感觉：[明确 / 稳定 / 有力量 / 克制]
- 次按钮应该感觉：[退后一步但仍清晰]
- 不要出现：[过度渐变 / 夸张阴影 / 玩具感]

### 表单类
- Input / Select / Textarea 的重点是：[清楚 / 易扫读 / 易纠错]
- Focus 态要：[明显但不刺眼]
- 错误提示要：[就近 / 明确 / 不靠猜]

### Card / List / Table
- 卡片用于：[分组信息 / 承载模块 / 展示摘要]
- 列表和表格优先：[信息清晰 / 对齐稳定 / 可快速扫描]
- 不要为了好看牺牲：[对齐 / 对比 / 可读性]

### Modal / Drawer
- 适合承载：[轻任务 / 配置项 / 二级流程]
- 不要把完整主流程都塞进弹层

### Navbar / Sidebar
- 导航要让用户一眼知道：[我在哪 / 我能去哪 / 当前重点是什么]
- 强调方式优先：[位置、字重、底色、边框]

## 5. 布局与留白

### 页面结构
- 页面宽度倾向：[居中窄栏 / 中宽内容区 / 大屏仪表盘]
- 内容组织方式：[单列 / 双列 / 栅格 / 模块拼装]
- 首页或主页面优先突出：[核心行动 / 核心数据 / 核心价值]

### 留白策略
- 区块之间：[疏 / 中 / 密]
- 模块内部：[紧凑 / 平衡 / 呼吸感强]
- 移动端优先保留：[主流程 / 关键 CTA / 关键状态]

## 6. 状态与动效

### 必须覆盖的状态
- Loading：[骨架屏 / spinner / 进度条 / 占位卡片]
- Empty：[解释为什么空 + 下一步引导]
- Error：[明确错误 + 恢复办法]
- Disabled：[让人知道为什么不能点]
- Hover / Focus / Active：[必须有清楚反馈]

### 动效原则
- 动效主要用于：[引导注意力 / 说明状态变化 / 降低切换生硬感]
- 动效节奏：[克制 / 自然 / 不抢内容]
- 不要出现：[炫技动画 / 无意义飘动 / 到处都在动]

## 7. Do / Don't

### Do
- Do: 优先复用已有 token 和组件
- Do: 保持统一圆角、阴影、边框、间距语言
- Do: 先保证清晰，再追求风格
- Do: 所有页面都补齐成功态之外的状态

### Don't
- Don't: 随机混用多种视觉流派
- Don't: 没依据时擅自加大面积炫技背景
- Don't: 用花哨样式掩盖信息结构不清
- Don't: 只做默认成功态，不做空态和错误态

## 8. 给 Agent 的执行提示

- 如果 `Spec`、技术 `Design`、`DESIGN.md` 冲突，优先级是：
  1. 产品边界和验收标准
  2. 技术约束
  3. 视觉细节
- 如果没有明确参考，优先做出统一、克制、可复用的页面，不要擅自炫技
- 如果已有 Figma，优先遵守 Figma，再用本文件补整体气质、状态页和缺失页面
- 如果发现当前页面会破坏全站一致性，优先回到已有组件和 token，而不是单页特殊处理
```

## 使用建议

推荐搭配下面几份文档一起用：

- `Proposal`：决定这次做什么、不做什么
- `Spec`：定义页面、流程、状态和验收标准
- 技术 `Design`：定义实现方案和边界
- `DESIGN.md`：定义视觉执行规则

推荐链路：

`Proposal -> Spec -> 技术 Design -> DESIGN.md -> Tasks -> 实现 -> QA`

## 一句话原则

**没有视觉边界，AI 就会靠猜；`DESIGN.md` 的作用就是把这种猜测压到最低。**
