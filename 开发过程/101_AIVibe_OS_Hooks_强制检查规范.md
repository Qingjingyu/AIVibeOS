# AIVibe OS Hooks 强制检查规范

## 这份文档解决什么问题

AIVibe OS 以前已经有很多“应该做”的规则。

问题是，只靠文档提醒，最后总会漏：

- 漏跑测试
- 漏补格式化
- 漏看高风险命令
- 漏留交付证据

这份文档的目标，是把这些“应该做”逐步升级成“自动检查”。

---

## 一、为什么 hooks 现在必须进主干

AI coding 走到现在，一个很明显的事实是：

**软提醒不够。**

真正影响稳定性的，往往是这些小动作：

- 改完没跑测试
- 直接用了危险命令
- 发布前没补回滚
- 文档和代码已经漂了

所以 AIVibe OS 现在要正式把 hooks 当成系统的一层，而不是附属技巧。

---

## 二、AIVibe OS 里的 hooks 分 3 类

### 1. 危险动作拦截

负责：

- 在动作发生前拦住明显高风险行为

典型场景：

- `kill -9`
- 破坏性 git 命令
- 越权目录写入
- 未审查的生产动作

### 2. 改动后自动检查

负责：

- 改完马上补最小工程检查

典型场景：

- formatter
- lint
- targeted tests
- 类型检查

### 3. 交付前证据检查

负责：

- 防止“看起来做完了，但证据不完整”

典型场景：

- phase-check
- QA 清单
- release / rollback 记录
- closeout

---

## 三、优先级最高的 5 个 hook

### Hook 1：危险命令前置拦截

这是第一优先级。

因为它解决的是“别把仓库和环境搞坏”。

### Hook 2：按改动文件触发最小检查

例如：

- 改前端文件，跑前端最小检查
- 改后端文件，跑后端最小检查
- 改文档，跑索引 / 入口检查

### Hook 3：提交前检查关键证据

至少检查：

- 有没有 README / Spec / QA / Release 的基本证据

### Hook 4：发布前检查回滚

没有回滚方案，不允许发布动作往下走。

### Hook 5：收尾后固定产出治理结果

默认挂到：

- `phase-check`
- `tool-registry`
- `bootstrap-board`
- `closeout`

---

## 四、AIVibe OS 当前推荐的落地顺序

不要一口气做全。

先按这个顺序来：

1. 危险动作拦截
2. 改动后最小检查
3. 收尾固定动作
4. 提交前证据检查
5. 发布前回滚检查

原因很简单：

**先把高频、高风险、最容易漏的地方接上。**

---

## 五、hooks 和脚本怎么分工

### hooks 负责

- 自动触发
- 在正确时机卡住流程

### scripts 负责

- 真正去执行检查
- 产出文件
- 产出报告

所以最推荐的方式不是把复杂逻辑都写进 hook。

而是：

**hook 负责触发，script 负责干活。**

---

## 六、现在最该先接哪几条

结合 AIVibe OS 当前状态，最该先接的是：

1. 文档治理 hook
2. 改动后最小测试 hook
3. 固定收尾 hook

因为这三条能最直接减少“系统看起来很完整，但实际没留证”的问题。

---

## 七、最容易犯的错

### 1. 一上来就做全套复杂 hooks

维护成本会太高。

### 2. hook 里塞太多业务逻辑

脚本更适合承载复杂逻辑。

### 3. 只有 hook，没有失败后的人工处理规则

最后团队会选择绕过。

### 4. 不区分本地检查和发布检查

会把日常效率拖死。

---

## 八、AIVibe OS 的最低标准

以后只要某条规则满足下面 3 条，就优先考虑升级成 hook：

1. 高频
2. 高风险
3. 经常漏

---

## 一句话铁规矩

**AIVibe OS 里，必须做的动作不能长期只靠记忆；能变成 hooks 的，迟早要变成 hooks。**
