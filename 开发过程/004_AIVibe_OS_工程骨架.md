# AIVibe OS 工程骨架

## 目标

这份文档回答一个最实际的问题：以后新产品开工时，仓库应该怎么搭，前端、后端、AI 和共享能力怎么分家。

目标不是一步到位做大平台，而是先把边界定清楚，保证后面能继续拆、继续扩。

## 推荐结构

建议默认采用 monorepo：

```text
apps/
  web/          前端应用
  api/          后端服务
  worker/       异步任务与长流程
packages/
  ui/           设计令牌与基础组件
  shared/       类型、常量、工具函数
  ai/           模型调用、提示词、评测、缓存
  config/       lint、tsconfig、环境配置模板
docs/
  specs/
  designs/
  qa/
```

## 分层原则

### 1. web

只负责页面、交互、状态展示和用户输入。

不要把模型调用、复杂业务规则、数据库操作直接塞进前端。

### 2. api

只负责业务接口、权限、数据读写和对外服务编排。

不要把页面逻辑和一次性脚本堆进 API 服务。

### 3. worker

负责耗时任务、异步流程、重试和补偿。

例如：

- 批量生成
- 文件处理
- AI 长任务
- 通知发送

### 4. packages/ui

负责设计令牌、基础组件和通用页面模块。

目标是让 Figma 资产先沉到这里，再被具体产品复用。

### 5. packages/ai

统一管理：

- 模型选择
- 提示词版本
- 调用日志
- 缓存
- 降级策略
- 评测入口

业务代码不应直接各自调用模型。

## 默认技术约束

- 前后端接口先定合同，再并行开发
- 所有跨端字段从 `shared` 出口统一
- AI 能力统一从 `packages/ai` 出口暴露
- 长任务默认走 `worker`，不阻塞主请求
- 日志、监控、错误码、权限规则统一沉到底座

## 一句话原则

**业务写在业务层，复用能力沉到底座，AI 调用统一收口。**
