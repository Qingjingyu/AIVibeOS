# AIVibe OS 自举与治理入口

## 这份文档解决什么问题

AIVibe OS 现在已经不只是“有很多规则”，而是已经开始有：

- 自我优化 Spec
- Top能力 规范
- Phase 证据化
- 实战回灌 SOP
- 工具总表
- 脚本化治理输出
- Skill 标准
- Command 映射
- 反偷懒机制

问题也正因为这样开始出现：

- 这些东西都重要，但第一次看的人不知道先看哪份
- 知道有治理脚本，但不知道什么时候跑
- 看到治理输出了，但不知道哪些要回写到系统文档

一句话：

**这一页负责把“自举与治理”这一组文档串成一条能执行的路径。**

## 二、每份文档分别负责什么

### 1. `082`

负责：

- 这轮为什么要优化 AIVibe
- 优先补哪几条主干
- 这一轮做到什么算过线

它是总方向，不是执行手册。

### 2. `084`

负责：

- Top能力 应该长什么样
- 以后能力包怎么统一结构

当治理动作涉及能力包时，一定要回来看它。

### 3. `085`

负责：

- 每个 Phase 到底凭什么说完成
- 证据应该长什么样

这份文档的作用，是把“感觉差不多”压回去。

### 4. `086`

负责：

- 一轮真实任务做完后，怎么把经验回灌回来

它管的是“做完之后怎么补系统”。

### 5. `087`

负责：

- 当前有哪些工具、脚本、能力包值得长期记住

它是武器总表。

### 6. `089`

负责：

- 哪些链路值得先脚本化
- 每个脚本的完成标准是什么

它是脚本化总方案。

### 7. `090`

负责：

- AIVibe 自己现在做到哪了
- P1 / P2 / P3 各自状态是什么

它是状态板，不是方法论。

### 8. `097`

负责：

- 每轮收尾时固定跑什么
- 输出会落到哪里
- 看完结果后要补什么

它是固定动作说明。

### 9. `103`

负责：

- Skill 到底是什么
- 什么样的工作流才值得正式化

### 10. `104`

负责：

- 命令如何触发 Skill
- 阶段入口如何和工作流绑定

### 11. `105`

负责：

- Skill 的 anatomy 怎么统一写
- 哪些字段以后必须有

### 12. `106`

负责：

- 怎么把“不要偷懒”变成系统机制
- 怎么把验证写进 Skill 和交付里

## 四、推荐固定动作

以后每轮补 AIVibe OS，建议最后固定做这 3 件事：

1. 跑一次固定收尾命令
2. 看一眼 closeout 总快照
3. 只把真正稳定的结论回写到母文档

固定命令：

```bash
bash ./scripts/governance/run-aivibe-closeout.sh
```

---

## 五、最容易犯的错

### 1. 只补文档，不跑治理脚本

这样系统状态还是假的。

### 2. 只看治理输出，不回写母文档

这样结果会越积越多，但系统主干不会变硬。

### 3. 把状态板当方法论文档

`090` 负责汇报状态，不负责解释全部打法。

### 4. 做完一轮不挂入口

这样下一轮还会重新找。
