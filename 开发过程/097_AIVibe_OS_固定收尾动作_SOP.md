# AIVibe OS 固定收尾动作 SOP

## 这份文档解决什么问题

AIVibe OS 现在已经有几条治理脚本了。

如果每次做完还靠人自己想：

- 这轮要不要跑脚本
- 先跑哪个
- 结果去哪里看
- 看完之后要不要回写

那这套系统还是会重新滑回“会做，但做不稳”。

一句话：

**这份 SOP 就是把每轮收尾动作钉死。**

---

## 一、固定收尾命令

以后每轮补 AIVibe OS 自己，默认执行：

```bash
bash ./scripts/governance/run-aivibe-closeout.sh
```

这条命令会顺序完成：

1. 跑 Phase 检查
2. 刷新工具台账表格
3. 刷新自举看板快照
4. 生成一份总收尾快照

---

## 二、默认输出会落到哪里

执行后，默认更新下面 4 个入口：

1. `治理输出/phase-check/latest.txt`
2. `治理输出/tool-registry/latest-table.md`
3. `治理输出/bootstrap-board/latest.md`
4. `治理输出/closeout/latest.md`

如果你只想看一份，先看第 4 个。

---

## 三、收尾后怎么判断结果

### 先看 `closeout/latest.md`

它会把这轮最重要的几份结果收成一页。

你先确认：

- Phase Summary 有没有退化
- 自举看板状态有没有变化
- 有没有新增缺项动作

### 再决定要不要回写母文档

只有下面这类东西，才值得回写：

- 稳定结论
- 新规则
- 新入口
- 新脚本说明

如果只是一次性结果，留在 `治理输出/` 就够了。

---

## 四、什么时候必须跑

下面 4 种情况，固定要跑：

1. 补完一轮 AIVibe OS 文档后
2. 新增或修改治理脚本后
3. 调整入口结构后
4. 准备对外汇报当前系统状态前

---

## 五、收尾后的人工动作

脚本能做的，是把结果收集出来。

脚本不能替你做的，是下面这 3 件事：

1. 判断哪些结果值得升格成母文档内容
2. 判断哪些只是一次性快照
3. 判断下一轮最该补哪里

所以固定顺序应该是：

`跑脚本 -> 看 closeout -> 再回写真正稳定的东西`

---

## 六、失败时怎么处理

如果固定收尾命令失败，不要直接跳过。

先看：

1. 是哪一条子脚本失败
2. 失败是路径问题、参数问题，还是脚本本身脏状态
3. 修完后重新跑完整条命令

原则很简单：

**收尾脚本失败时，不要只修一半就宣称本轮结束。**
