# AIVibe OS Skill Anatomy 模板

## 这份文档解决什么问题

如果不同 Skill 各写各的，后面一定会出现：

- 有的只有步骤
- 有的只有理念
- 有的没有边界
- 有的没有验证

结果就是：

**Skill 看起来很多，但质量完全不齐。**

所以 AIVibe OS 需要一个统一的 Skill Anatomy。

---

## 一、统一结构

以后任何正式 Skill，默认按下面结构写。

```md
# Skill Name

## 解决什么问题

## 什么时候用

## 不什么时候用

## 输入

## 输出

## 执行步骤

## 常见偷懒借口

## 红线与危险信号

## 验证方式

## 交接产物

## 真实例子
```

---

## 二、每部分分别写什么

### `解决什么问题`

一句话讲清这个 Skill 的目标。

### `什么时候用`

列出适用场景。

### `不什么时候用`

列出禁用场景。

### `输入`

写清必须拿到什么材料。

### `输出`

写清必须交回什么结果。

### `执行步骤`

按顺序写，不要写成散点建议。

### `常见偷懒借口`

把最常见的跳步理由写出来。

例如：

- “先做出来再补测试”
- “这次只是小改动”
- “文档以后再写”

### `红线与危险信号`

写清什么情况下必须停下来。

### `验证方式`

说明如何证明这个 Skill 真被执行过、结果合格。

### `交接产物`

说明交给下一个人的最低材料。

### `真实例子`

最好放真实案例，不要只写抽象说明。

## 三、最小可复制模板

```md
# <Skill Name>

## 解决什么问题

## 什么时候用

## 不什么时候用

## 输入

## 输出

## 执行步骤
1.

3.

## 常见偷懒借口

## 红线与危险信号

## 验证方式

## 交接产物

## 真实例子
```

---

## 四、为什么一定要有“偷懒借口”

因为 AI 和人都会自动找最短路径。

如果 Skill 里不提前写明最常见的偷懒借口，实际执行时很容易回到：

- 跳 Spec
- 跳测试
- 跳 Review
- 跳文档

所以这不是附加项，而是 AIVibe OS Skill Anatomy 的核心部分。

---

## 五、为什么一定要有“验证方式”

没有验证，Skill 只是“看起来像执行过”。

有了验证，Skill 才有证据。

验证可以是：

- 测试结果
- 截图
- 变更说明
- QA 记录
- release 记录

---

## 六、最容易犯的错

### 1. 把 Anatomy 写成大而空的理念文

Agent 没法执行。

### 2. 只有步骤，没有输入输出

前后衔接会断。

### 3. 只有验证，没有偷懒借口

AI 还是会选择最快路径。

### 4. 只有模板，没有真实例子

别人上手成本会很高。

---

## 一句话铁规矩

**AIVibe OS 的 Skill Anatomy 必须同时解决：怎么做、什么时候停、怎么证明做过。**
