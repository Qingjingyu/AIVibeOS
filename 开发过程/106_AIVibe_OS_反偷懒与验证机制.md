# AIVibe OS 反偷懒与验证机制

## 这份文档解决什么问题

AI coding 最大的问题之一，不是不会做，而是总想走最短路径。

典型表现就是：

- 先写代码，后补需求
- 先改完，后补测试
- 先上线，后补回滚
- 先说完成，后补证据

这份文档的目标，就是把 AIVibe OS 里的“反偷懒”和“验证”变成正式机制。

---

## 一、什么叫偷懒

在 AIVibe OS 里，偷懒不是工作快。

偷懒指的是：

- 跳过必要步骤
- 缺证据就宣称完成
- 明知道有风险还不补边界
- 用“这次先这样”替代正式做法

一句话：

**偷懒不是省力，而是把风险留给后面。**

---

## 二、最常见的 8 种偷懒借口

1. 这只是个小改动
2. 先做出来再补 Spec
3. 先不写测试，肉眼看过了
4. 这次先不做回滚
5. 先不补文档，后面统一整理
6. 这块应该很简单，不需要 Review
7. 现在先快一点，后面再重构
8. 差不多能跑，就算完成

以后只要听到类似说法，就该警觉。

---

## 三、AIVibe OS 的反偷懒机制分 4 层

### 第 1 层：命令前置

通过 `/spec /plan /build /test /review /ship` 这样的阶段命令，强迫流程按顺序走。

### 第 2 层：Skill Anatomy

通过“常见偷懒借口”和“验证方式”，在 Skill 内部提前设防。

### 第 3 层：hooks / scripts

把高频漏项做成自动检查。

### 第 4 层：交付证据

没有测试、截图、QA、release 记录，就不能说完成。

---

## 四、最小验证清单

以后一个正式任务，最少应有下面这些验证里的若干项：

- 需求文档或 Spec
- 任务拆解
- 测试结果
- 页面截图或运行结果
- QA 清单
- Release / Rollback 记录
- 差异说明

不是每次都要全部齐，但正式交付一定不能几乎没有证据。

---

## 五、什么时候必须停下来

下面这些情况，默认必须停下来，而不是继续硬冲：

1. 连需求边界都说不清
2. 连最小验证方式都没有
3. 同一个问题已经失败 3 次
4. 高风险操作没有回滚
5. 交付结果无法证明

这类情况继续做，只会加大返工。

---

## 六、AIVibe OS 的默认回答方式

以后只要系统发现自己在走最短路径，就要主动回到这三个问题：

1. 这一步是不是跳过了某个必要环节
2. 现在有什么证据能证明结果有效
3. 如果失败了，怎么回退

只要有一个答不上来，就不该轻易往下宣称完成。

---

## 七、怎么把这套机制接到系统里

最推荐的接法是：

- Command 负责阶段约束
- Skill 负责反偷懒提醒
- Hook 负责自动拦截
- QA / Release 负责最终验收

这四层一起用，反偷懒才真正有效。

---

## 八、最容易犯的错

### 1. 只讲“不要偷懒”

没有用，必须写成机制。

### 2. 只讲验证，不讲阶段边界

最后会经常补证据补到一半才发现方向错了。

### 3. 只讲机制，不讲真实借口

AI 和人都会找到新说法继续跳步。

### 4. 把“能跑”当“完成”

这是最危险的一种偷懒。

---

## 一句话铁规矩

**AIVibe OS 不是靠“自觉不偷懒”来保证质量，而是靠阶段命令、Skill、hooks 和证据一起把偷懒空间压缩掉。**
