# AIVibe OS 指令栈与规则分层规范

## 这份文档解决什么问题

AI coding 走到现在，最大的问题已经不是“规则够不够多”，而是：

- 总规则写在哪
- 工具规则写在哪
- 模块规则写在哪
- 任务规则写在哪
- 哪些规则只是建议，哪些是硬约束

如果这些不分层，结果通常只有一个：

**规则越写越多，冲突也越多。**

---

## 一、AIVibe OS 的默认指令栈

以后 AIVibe OS 默认按这 5 层组织规则。

### 第 1 层：仓库总规则

默认入口：

- `AGENTS.md`

负责：

- 统一世界观
- 统一协作方式
- 统一硬边界

它不应该塞满目录细节和一次性任务说明。

### 第 2 层：工具专属适配

例如：

- `CLAUDE.md`
- `.claude/`
- `.cursor/rules/`
- GitHub custom agent 配置
- Codex / agent-specific config

负责：

- 把总规则翻译成某个工具能吃的形式
- 补工具自己的能力入口

它不应该重新定义产品和架构原则。

### 第 3 层：路径 / 模块规则

例如：

- `apps/web/AGENTS.md`
- `packages/ai/AGENTS.md`
- 某个目录下的专属 rules 文件

负责：

- 说明这个模块的边界
- 说明本目录允许和禁止的改法

### 第 4 层：任务级规则

例如：

- proposal
- spec
- design
- tasks
- plan 文档

负责：

- 这次任务为什么做
- 做到哪里算完成
- 这次只改什么，不改什么

### 第 5 层：运行时硬约束

例如：

- hooks
- scripts
- CI
- lint / test / release 检查

负责：

- 把必须执行的动作变成硬门槛

---

## 二、每层分别该写什么

### `AGENTS.md` 里应该有

- 项目目标
- 协作方式
- 核心流程
- 质量底线
- 高风险禁区

### 工具适配层应该有

- 如何导入总规则
- 工具专属能力入口
- 工具专属限制

### 模块层应该有

- 本目录职责
- 上下游边界
- 常见禁改点
- 本目录最常用命令

### 任务层应该有

- 背景
- 范围
- 验收标准
- 当前决策

### hooks / 脚本层应该有

- 什么时候触发
- 触发后跑什么
- 失败后怎么处理

---

## 三、最推荐的目录结构

```text
repo/
  AGENTS.md
  CLAUDE.md
  .claude/
    rules/
    agents/
  .cursor/
    rules/
  apps/
    web/
      AGENTS.md
    api/
      AGENTS.md
  docs/
    specs/
    plans/
```

重点不是目录长什么样，而是：

**总规则、工具适配、模块边界、任务计划、运行时检查，必须是五层。**

---

## 四、AIVibe OS 当前的落地原则

### 原则 1：总规则只写一次

总世界观默认只在 `AGENTS.md` 定义。

其他地方尽量引用，不要复制。

### 原则 2：工具层只做翻译

工具适配文件的任务是“接通”，不是“改宗”。

### 原则 3：模块层优先说边界

不要在模块规则里重复大段流程说明。

模块规则最重要的是：

- 这里负责什么
- 这里不负责什么

### 原则 4：任务层必须短命

任务 spec / plan 是为这轮任务服务，不要把一次性决策污染总规则。

### 原则 5：硬约束尽量放到运行时

能让 hooks / 脚本 / CI 自动做的，尽量不要只写成提醒。

---

## 五、最容易犯的错

### 1. 每层都写一遍总规则

最后一定会冲突。

### 2. 把模块细节塞进总规则

总规则会越写越臃肿。

### 3. 把任务级临时决定回写成永久规则

系统会越来越僵硬。

### 4. 只有文档，没有运行时硬约束

这样规则永远只是“希望”。

---

## 六、一条最简单的判断法

如果一句规则离开这次任务、下个季度还应该成立，它大概率属于上层。

如果一句规则只对这次工作成立，它应该留在任务层。

---

## 一句话铁规矩

**AIVibe OS 的规则必须分层：总规则定方向，工具层做适配，模块层定边界，任务层定本轮，hooks 层兜底。**
