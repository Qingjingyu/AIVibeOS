# AIVibe OS 自我优化 Spec v1

## 这份文档解决什么问题

AIVibe OS 现在已经不是“没东西”。

问题恰恰相反：

- 已经有很多方法
- 已经有很多模板
- 已经有很多专题结论
- 也已经开始有真实项目验证

但如果继续按现在的方式往前长，很容易出现一个新问题：

**资产越来越多，真正可执行、可复用、可持续升级的主干反而不够清楚。**

这份 Spec 的目标，就是把 AIVibe OS 下一阶段该怎么优化自己，说清楚。

一句话：

**从“会写很多对的文档”，升级成“能稳定驱动真实项目落地的操作系统”。**

---

## 一、当前状态判断

先不客气，直接下判断。

### 现在做得好的地方

1. 方法层已经很强  
   已经有开发流程、Spec、Figma、AI Runtime、QA、发布、回滚这些母文档。

2. 研究吸收能力已经建立起来  
   外部项目不是简单摘抄，而是能提炼成内部规则。

3. 真实项目验证已经开始  
   尤其是 Figma to Code 这条线，已经不只是停留在纸面。

4. 资产沉淀意识已经形成  
   每轮做完都会尽量落文档、模板、记录、结论。

### 现在最明显的短板

1. 入口还是偏散  
   新人或者未来的自己进来，还是容易“看见很多文档，但不知道现在先用哪几个”。

2. 关键能力还没有彻底产品化  
   比如 `Figmatocode` 已经很强，但还更多是“能力包”，还不是“自动机器”。

3. Phase 进度还偏主观  
   现在很多“完成了没”，还是靠判断，不是靠证据板。

4. 实战回灌机制还不够硬  
   做完一个真实项目后，经验会沉淀，但沉淀动作还不够像流水线。

5. 工具资产分布还不够统一  
   skill、脚本、模板、参考仓、MCP 能力都在，但还没形成一眼就懂的总台账。

---

## 二、核心结论

AIVibe OS 下一阶段不应该继续把主要精力放在：

- 再写更多同类型原则文档
- 再补很多相似模板
- 再扩大“知识库存量”

而应该把主战场切到下面 5 件事：

1. **统一入口层**
2. **Top 能力标准化**
3. **Phase 证据化**
4. **实战回灌流水线**
5. **工具与依赖登记层**

这 5 件事做好，AIVibe OS 才会从“强脑子”变成“强执行系统”。

---

## 三、优化目标

### 目标

在不推翻现有资产的前提下，让 AIVibe OS 变成：

- 新项目能快速上手
- 老项目能持续复用
- 每轮实战都能自动沉淀
- 每个 Phase 都有证据
- 每个核心能力都能独立打包、交接、复用

### 非目标

这一轮不追求：

- 一次性重写整个仓库
- 把所有历史文档重新整理一遍
- 立刻做成完整平台产品
- 追求表面“整齐”，牺牲实用性

原则很简单：

**先补主干，不搞大扫除式重构。**

---

## 四、优化范围

这一轮优化，分成 P1、P2、P3。

### P1：必须先做

### 1. 统一入口层

目标：

让任何一个人进入 AIVibe OS 后，能在 3 分钟内知道：

- 我现在在哪个阶段
- 我应该先看什么
- 我这次要用哪套能力
- 我最终应该产出什么

要补的东西：

- 一个真正的“总入口页”
- 一个按场景使用的导航页
- 一个按阶段使用的导航页
- 一个“新项目启动最小路径”

验收标准：

- 第一次进仓的人，不需要翻十几份文档找入口
- 不同任务类型都能找到最短阅读路径

### 2. Top 能力标准化

目标：

把 `Figmatocode` 这种能力包，从“做得不错的样板”升级成“统一模板”。

要补的东西：

- Top 能力的统一目录规范
- Top 能力必须具备的最小文件清单
- Top 能力的验收规则
- Top 能力的交接规则

验收标准：

- 以后新增一个 Top 能力时，不再临时决定结构
- 同事拿到任意 Top 能力包，都知道怎么用

### 3. Phase 证据化

目标：

把“Phase 完成没”从感觉判断，变成证据判断。

要补的东西：

- 每个 Phase 的完成条件
- 每个 Phase 对应要留下的证据
- 一份统一的 Phase 看板模板

证据例子：

- 研究结论
- 实战链接
- 截图对稿结果
- 回放记录
- QA 结果
- 发布记录

验收标准：

- 以后说一个 Phase 已完成，必须拿得出对应证据

### 4. 实战回灌流水线

目标：

把“做完一个真实项目后顺手写点文档”，升级成固定动作。

要补的东西：

- 实战结束后的标准回灌步骤
- 回灌要写哪几份文档
- 结论怎么回写到母文档
- 哪些东西要沉淀成 Top 能力

验收标准：

- 以后每做完一轮真实项目，都能反向增强 AIVibe OS
- 不再靠记忆补文档

### 5. 工具与依赖登记层

目标：

把用过的 MCP、skill、参考仓、脚本、浏览器工具、验收工具，全部纳入统一台账。

要补的东西：

- 工具总表
- 每个工具的用途
- 使用前提
- 成功案例
- 风险和替代方案

验收标准：

- 以后不会再出现“这个工具之前用过，但不知道在哪”的情况

### P2：第二阶段做

### 1. 模板减负

把明显重复、重叠、低使用率的模板做一次合并。

目标不是减少数量本身，而是减少选择负担。

### 2. 核心链路脚本化

优先把重复动作变成脚本或命令，比如：

- Figma 拉取
- 截图对稿
- 页面验收
- 资产收集

### 3. AIVibe 自举看板

把“我们正在优化 AIVibe 自己”的状态也做成可追踪看板。

### P3：第三阶段做

### 1. Top 能力产品化

把 `Figmatocode` 这种能力，进一步推进成半自动或全自动产品。

### 2. 模板代码化

把部分高频模板升级成可以直接生成项目骨架的脚手架。

### 3. 团队协作分发机制

让 Top 能力可以直接打包、发放、版本升级、统一维护。

## 五、建议的结构补强图

当前最需要的，不是再横向铺新文档，而是把主干补粗。

建议按这个结构加强：

```text
AIVibe OS
├── 总入口
│   ├── README
│   ├── 总索引
│   ├── 场景导航
│   └── 阶段导航
├── 母文档层
│   ├── 流程
│   ├── 工程骨架
│   ├── Figma
│   ├── AI Runtime
│   └── QA / 发布
├── Top能力层
│   ├── Figmatocode
│   ├── 后续能力A
│   └── 后续能力B
├── 实战记录层
│   ├── 项目记录
│   ├── 验收证据
│   └── 复盘结论
└── 工具登记层
    ├── MCP
    ├── Skill
    ├── 脚本
    └── 外部参考仓
```

核心思想：

**母文档讲原则，Top 能力讲打法，实战记录讲证据，工具登记讲抓手。**

---

## 六、执行顺序

建议按下面顺序优化，不要乱。

### 第一步：先把入口做顺

因为入口不顺，后面所有能力都会显得难用。

### 第二步：把 Top 能力标准化

因为这是最容易直接产生复用价值的地方。

### 第三步：把 Phase 证据化

因为没有证据，路线图容易变成自我感觉良好。

### 第四步：把实战回灌固定下来

因为 AIVibe OS 的真正增长，不来自空想，而来自实战后的反哺。

### 第五步：再做工具台账和自动化

因为只有先知道自己有哪些武器，自动化才不会乱接。

## 七、这份 Spec 对应的具体交付物

如果按这个 Spec 执行，建议最终补出下面几类东西。

### 入口类

- 一份“新项目从 0 到 1 使用路径”
- 一份“按任务场景选文档”的导航页
- 一份“按 Phase 选动作”的导航页

### 标准化类

- 一份 `Top能力` 统一规范
- 一份 `Top能力` 最小验收清单
- 一份 `Top能力` 交接模板

### 证据类

- 一份 Phase 证据模板
- 一份 AIVibe 自举状态板

### 回灌类

- 一份真实项目回灌 SOP
- 一份“哪些内容回写母文档，哪些内容沉淀 Top 能力”的判断规则

### 台账类

- 一份工具与依赖总表
- 一份工具成功案例记录模板

---

## 八、验收标准

这一轮优化做完，不是看文档数量，而是看下面 7 条能不能成立。

1. 新人进来，知道先看什么
2. 新项目启动，知道先产出什么
3. 核心能力包，知道怎么复用
4. 每个 Phase，知道凭什么算完成
5. 真实项目做完，知道怎么回灌
6. 常用工具，知道去哪里找
7. AIVibe OS 自己，开始形成持续自我增强闭环

只要这 7 条里大部分还做不到，就说明优化还没到位。

---

## 九、风险提醒

这一轮最容易犯的错有 4 个。

### 1. 又写成一堆新文档

如果只是在现有基础上继续横向加文件，而不是补主干，那问题会更严重。

### 2. 想一步做到完美

这会把“补系统主干”重新拖回“大而全整理工程”。

### 3. 只讲方法，不做证据层

没有证据，AIVibe OS 还是会停留在“懂很多道理”。

### 4. 只做文档，不做真实回灌

这会让系统再次脱离真实使用场景。

---

## 十、一句话版本

AIVibe OS 下一阶段最重要的，不是继续把自己写得更大，而是把自己变得更顺、更硬、更能自我进化。

一句铁话：

**先做统一入口，再做能力标准化，再做证据闭环，最后做自动化放大。**
