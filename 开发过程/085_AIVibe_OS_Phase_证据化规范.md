# AIVibe OS Phase 证据化规范

## 这份文档解决什么问题

以前我们说一个 Phase “差不多完成了”，很多时候靠的是感觉。

这会带来 3 个问题：

1. 进度容易虚高
2. 回头复盘时没有依据
3. 不同人对“完成”的理解不一样

这份文档的目标，就是把每个 Phase 的完成，变成有证据的完成。

一句话：

**以后 AIVibe OS 里，Phase 不是靠嘴说完成，而是靠证据说完成。**

---

## 一、什么叫 Phase 证据化

Phase 证据化，就是每进入一个阶段，都要留下能复查的东西。

这些东西至少要回答：

- 这阶段做了什么
- 结果是什么
- 质量怎么样
- 下一阶段凭什么能继续

如果没有证据，最多只能说“正在做”，不能说“做完了”。

---

## 二、证据分 4 类

为了避免每次都乱记，AIVibe OS 统一把证据分成 4 类。

### 1. 决策证据

用来证明：

- 为什么这么做
- 为什么不那么做

常见形式：

- Proposal
- Spec
- 设计决策
- 取舍说明

### 2. 执行证据

用来证明：

- 事情真的做了
- 不是停在讨论层

常见形式：

- 代码链接
- 页面截图
- 配置截图
- 接口返回
- 任务完成记录

### 3. 质量证据

用来证明：

- 做出来的东西不是假完成

常见形式：

- 测试结果
- QA 记录
- 截图对稿
- 差异说明
- 回放记录

### 4. 运维证据

用来证明：

- 这个东西可以上线、可回滚、可继续维护

常见形式：

- 发布记录
- 监控截图
- 回滚方案
- 事故记录

---

## 三、每个 Phase 最少要留什么证据

### Phase 1：Discovery

最少证据：

- 研究输入
- 目标用户判断
- MVP 与非 MVP 划分
- 风险清单

通过标准：

- 方向清楚
- 范围收住

### Phase 2：Planning

最少证据：

- Proposal
- Spec
- Design
- Tasks

通过标准：

- 要做什么已经写清
- 不做什么也写清

### Phase 3：Building

最少证据：

- 代码或文档实现结果
- 页面或接口截图
- 最小验证结果
- 已完成任务记录

通过标准：

- 不是 PPT 完成
- 关键路径能跑

### Phase 4：Polish

最少证据：

- 视觉修正记录
- 边界场景处理结果
- 多端或多状态验证结果
- 性能或体验修正说明

通过标准：

- 不是只有主流程能看
- 边角不再明显掉链子

### Phase 5：Handoff

最少证据：

- README 或使用说明
- 交接说明
- 发布前检查结果
- 后续建议

通过标准：

- 别人能接手
- 不是只能作者本人维护

## 四、统一的 Phase 状态定义

以后每个 Phase 只允许用下面 4 个状态。

### `未开始`

还没有进入这个阶段。

### `进行中`

已经开始做，但证据还没补齐。

### `待验收`

主要内容已完成，但还没完成检查和确认。

### `已完成`

证据齐全，且已经通过本阶段验收。

不要再用这些模糊词：

- 差不多
- 基本好了
- 先算完成
- 后面再补

这些词会直接把系统搞乱。

## 六、AIVibe OS 自己的证据要求

AIVibe OS 不是例外。

以后优化 AIVibe OS 自己，也必须按这套规则来。

比如：

- 新增一份母文档，要有为什么新增的依据
- 新增一份 Top能力，要有真实验证
- 说某个优化完成了，要拿得出对应入口、模板或记录

否则 AIVibe OS 就会重新掉回“会讲道理，但证据不硬”。

---

## 七、什么时候不算完成

下面这些情况，一律不算 `已完成`。

### 1. 只有文档，没有验证

说明还停在纸面。

### 2. 只有结果，没有过程证据

说明后面很难复盘和交接。

### 3. 只有主流程，没有边界验证

说明质量还不稳定。

### 4. 只有作者自己知道怎么继续

说明还没进入可交接状态。

---

## 八、使用建议

最推荐的用法是：

1. 项目启动时先建 Phase 看板
2. 每进入一阶段就补证据
3. 每次 Review 都只看证据，不听口头感觉
4. 阶段没过，不要硬进下一阶段

这样虽然前面稍微麻烦一点，但后面返工会少很多。

---

## 一句话铁规矩

**Phase 完成不靠主观感觉，靠证据闭环；没有证据，就没有完成。**
