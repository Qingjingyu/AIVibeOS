# AIVibe OS 质量与发布标准

## 目标

解决“功能做完了但不敢发”“上线后出问题查不到”“每次发版都靠经验”的问题。

这份标准的作用，是把“看起来能发”变成“有证据可以发”。

## 完成不等于上线

一个功能写完，只代表实现完成，不代表可以上线。

在 AIVibe OS 里，一个功能要进入发布，至少要同时满足：

- 需求已经验收
- 主流程和异常流程测过
- 日志和监控能看到
- 回滚办法清楚
- 风险和影响范围说清楚

## 最低质量要求

每个功能默认都要检查：

- 主流程是否可用
- 异常提示是否可理解
- 边界状态是否完整
- 多端表现是否正常
- 是否影响已有功能

如果是 AI 相关功能，还要额外检查：

- 输出稳定性
- 成本是否可控
- 模型异常时是否可降级

## 发布前必须准备的证据

- 对应的 proposal / spec / design
- 测试结果
- 页面或接口验证记录
- 关键日志截图或说明
- 发布步骤
- 回滚方案

谁说“可以发”，谁就要能拿出这些证据。

## 发布步骤原则

发布时遵守四件事：

1. 先备份关键数据和配置
2. 先发风险低的部分，再发核心链路
3. 发布后立刻验证页面、接口、日志和监控
4. 一旦异常超出预期，立即回滚，不赌运气

## 回滚不是可选项

每次发布前都必须提前讲清楚：

- 什么情况下回滚
- 回滚谁来做
- 回滚怎么做
- 回滚后怎么验证恢复正常

没有回滚方案，不算准备完成。

## 一条铁规矩

**没有测试结果、没有监控入口、没有回滚方案的改动，不允许上线。**
