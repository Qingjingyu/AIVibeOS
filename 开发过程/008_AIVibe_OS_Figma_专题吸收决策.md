# AIVibe OS Figma 专题吸收决策

## 这轮研究要吸收什么

基于 Figma 官方文档、设计 token 实践项目和 `web-access` 仓库，这轮专题后 AIVibe OS 先吸收下面 5 个决策。

## 决策 1：MCP 是默认入口

以后只要是 Figma 到代码的任务，默认先走 MCP：

- 取设计上下文
- 取变量
- 取组件映射
- 再决定复用还是新增

不再默认依赖截图硬还原。

## 决策 2：Code Connect 从高频组件开始

第一批只映射最常用组件：

- Button
- Input
- Card
- Modal
- Tabs

原因很简单：这些组件命中率最高，最能快速验证“设计组件和真实代码组件连起来”到底有没有价值。

## 决策 3：Tokens 必须进入代码构建链路

以后 tokens 不允许只停在 Figma 里。

标准方向定为：

`Figma Variables / Tokens -> JSON -> Style Dictionary -> packages/ui`

这样才能保证：

- 页面不用手写散落颜色和尺寸
- 多产品能共享视觉基础
- 改 token 时可以自动扩散到代码层

## 决策 4：`web-access` 作为研究和验证层接入

`web-access` 不参与主生成链路，但纳入研究工具栈。

它的职责是：

- 调研竞品或设计系统官网
- 观察动态交互和状态
- 辅助分析复杂页面行为

它解决的是“怎么看真实网页”，不是“怎么替代设计系统映射”。

## 决策 5：先验证一条最小闭环

第一条最小闭环不要贪大，建议这样跑：

1. 选一个 Figma 页面
2. 先抽 tokens
3. 再映射 3 到 5 个高频组件
4. 用这些组件拼出一个真实页面
5. 检查复用率、还原速度和维护成本

只有这条链路跑通，后面再扩大范围。
