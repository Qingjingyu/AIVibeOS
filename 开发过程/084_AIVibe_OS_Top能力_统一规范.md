# AIVibe OS Top能力 统一规范

## 这份文档解决什么问题

现在 AIVibe OS 里已经出现了第一份真正像样的 `Top能力`：

- `Top能力/Figmatocode/`

这很好。

但如果没有统一规范，后面每新增一个 Top 能力，都可能重新长一套结构。

结果就是：

- 文件名不统一
- 入口方式不统一
- 验收方式不统一
- 交接方式不统一
- 别人拿到后不知道先看什么

这份文档的目标，就是把 `Top能力` 变成标准件。

一句话：

**以后任何一个 Top 能力，都必须像一个能独立交付的小产品，而不是一堆散文件。**

---

## 一、什么是 Top能力

在 AIVibe OS 里，`Top能力` 不是普通文档夹。

它指的是：

- 已经被真实项目验证过
- 能独立打包
- 能独立交接
- 能直接复用
- 对外发给别人也能看懂、能上手

的核心能力包。

它比普通母文档更“落地”，比一次性复盘更“稳定”。

### 适合进入 Top能力 的内容

- 已完成真实项目验证的能力
- 高频复用的核心工作流
- 团队内多人都会重复用到的打法
- 可以独立交给 Agent 执行的能力

### 不适合进入 Top能力 的内容

- 还没验证过的想法
- 一次性项目记录
- 只有原则、没有动作的抽象总结
- 只有资料堆，没有执行方法的学习包

---

## 二、Top能力 的准入门槛

一个东西想进 `Top能力/`，至少要满足下面 6 条。

### 1. 有明确问题定义

必须说清楚：

- 它解决什么问题
- 适合什么场景
- 不适合什么场景

### 2. 有稳定流程

不能只是“经验感觉”。

必须能写成：

- 输入是什么
- 过程怎么走
- 输出是什么

### 3. 有真实验证

至少要有一次真实项目或真实页面验证。

没有实战，只能算草案，不能进 Top能力。

### 4. 有验收标准

不能只说“感觉不错”。

必须说清楚：

- 怎么判断做完
- 怎么判断合格
- 有什么最小证据

### 5. 有交接材料

别人接手时，不能靠口头解释。

至少要有：

- README
- Agent 用法
- 人类同事用法
- 交接方式

### 6. 能独立打包

把这个文件夹单独发给别人后，对方也能看懂和使用。

如果离开仓库上下文就不能用，说明还不算成熟。

---

## 三、Top能力 的统一目录

以后默认按下面结构来，不再临时发挥。

```text
Top能力/
  能力名/
    README.md
    SKILL.md
    01_完整SOP.md
    02_Agent_Prompt_模板.md
    03_交接话术模板.md
    04_验收清单模板.md
    CHANGELOG.md
    examples/
    doc/
```

### 每层负责什么

### `README.md`

总入口。

解决：

- 它是什么
- 解决什么问题
- 文件夹里有什么
- 先看什么

### `SKILL.md`

给 Agent 执行的版本。

要求：

- 说清输入
- 说清流程
- 说清输出
- 说清禁用动作
- 默认遵循 [105_AIVibe_OS_Skill_Anatomy_模板.md](./105_AIVibe_OS_Skill_Anatomy_模板.md)

### `01_完整SOP.md`

给人完整理解打法。

这是“母打法”，不是流水账。

### `02_Agent_Prompt_模板.md`

让团队能快速稳定发起任务。

### `03_交接话术模板.md`

保证中途换人、换 Agent、交外包时不失真。

### `04_验收清单模板.md`

保证完成不是靠感觉。

### `CHANGELOG.md`

记录这个能力包怎么变的。

### `examples/`

放真实案例。

要求：

- 不是假想例子
- 最好能看到输入、过程、结果

### `doc/`

放这个能力的黑匣子资料。

比如：

- 资源清单
- 产品规划
- 已知限制
- 自动化方向

---

## 四、Top能力 的最小文件清单

任何一个 Top能力，最少必须有下面 8 样东西。

1. `README.md`
2. `SKILL.md`
3. 一份完整 SOP
4. 一份 Agent Prompt 模板
5. 一份交接模板
6. 一份验收清单
7. 一份真实示例
8. 一份变更记录

缺任何一个，都不要急着对外打包。

补充：

如果这套 Top能力 还准备继续演进成正式 Skill 包，至少还要补：

- 对应的 Skill 标准
- 明确的 command 触发入口
- 反偷懒与验证机制

---

## 五、Top能力 的完成定义

一个 Top能力 算完成，不是“文件看起来够多了”。

至少要满足：

### 1. 能独立理解

第一次接触的人，看 `README` 就知道这套能力是干什么的。

### 2. 能直接执行

Agent 读 `SKILL.md` 能按步骤做事，而不是还要猜。

### 3. 能被真实复用

至少在一个真实项目或真实页面里用过。

### 4. 有验收证据

至少要能留下：

- 截图
- 链接
- 测试结果
- 差异说明

### 5. 有边界说明

必须说清“不适合什么”。

否则很容易被乱用。

### 6. Skill Anatomy 合格

如果里面有 `SKILL.md`，就不能只是散步骤。

至少要能回答：

- 什么时候用
- 不什么时候用
- 常见偷懒借口
- 怎么验证

---

## 六、Top能力 的交接规则

Top能力 对外交接时，默认要说清下面 5 件事。

1. 这套能力解决什么问题
2. 这次准备怎么用
3. 哪些输入必须准备好
4. 成功标准是什么
5. 当前已知限制是什么

如果交接时连这 5 件事都说不清，就说明这个能力还不够成熟。

---

## 七、Top能力 的验收规则

以后验收一个 Top能力，不要只看“写没写文档”。

按下面 4 层验：

### 第一层：结构是否完整

目录、入口、模板、示例、记录是否齐。

### 第二层：内容是否能执行

是不是说人话。

是不是给了明确动作。

是不是别人能照着做。

### 第三层：是否经过真实项目

没有真实用过，就不算通过。

### 第四层：是否能脱离作者本人

别人能不能接手。

能不能打包发出去。

能不能持续迭代。

---

## 八、Top能力 的推荐演进顺序

以后一个新能力，建议按下面顺序长，不要倒着来。

1. 先做真实任务
2. 再提炼动作
3. 再写 SOP
4. 再补 Prompt 和交接模板
5. 再补验收清单
6. 再放入 `Top能力/`
7. 最后再考虑自动化和产品化

不要一开始就搭大文件夹。

先证明这能力真有用。

---

## 九、现阶段的标准样板

当前 AIVibe OS 里，`Top能力/Figmatocode/` 就是第一份标准样板。

它证明了这条路是对的：

- 先实战
- 再抽方法
- 再补验收
- 再做打包

后面的 Top能力，原则上都应该向它看齐，但不允许机械复制。

要学的是结构和交付方式，不是文件名数量。

---

## 十、可直接复制的 Top能力 自检清单

在一个能力准备进入 `Top能力/` 前，直接过这 10 条。

- 它解决的问题是不是一句话能说清
- 它有没有至少一次真实验证
- 它有没有总入口 README
- 它有没有 Agent 可执行版本
- 它有没有完整 SOP
- 它有没有交接模板
- 它有没有验收清单
- 它有没有真实案例
- 它有没有变更记录
- 它是不是可以脱离作者本人继续用

只要有 2 条以上答不上来，就先别放进 `Top能力/`。

---

## 一句话铁规矩

**Top能力 不是“最重要的资料夹”，而是“已经被证明能独立交付的能力产品”。**
