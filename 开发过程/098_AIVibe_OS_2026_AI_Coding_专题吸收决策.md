# AIVibe OS 2026 AI Coding 专题吸收决策

## 这份文档解决什么问题

外部 AI coding 工具和教程已经长得很多、很快。

如果不做一次明确决策，AIVibe OS 很容易继续出现两种问题：

- 看了很多新东西，但系统主干没变硬
- 什么都想吸收，最后把系统边界搞乱

这份文档只做一件事：

**决定这轮到底吸收什么，不吸收什么。**

---

## 一、这轮的总判断

按 2026-04-22 当前状态，这一轮最值得吸收的，不是新的模型名，也不是某个工具的新按钮。

真正值得吸收的是 4 条已经稳定下来的系统级变化：

1. 指令栈分层
2. 多代理执行模式
3. hooks 驱动的硬约束
4. agent profile 标准化

一句话：

**AIVibe OS 要从“会写规则”升级成“会调度代理、会硬约束流程”的系统。**

---

## 二、这轮立刻吸收的 5 条决策

### 决策 1：把 `AGENTS.md` 定为跨工具总入口

以后跨工具不再各写一套完全不同的总规则。

默认做法：

- `AGENTS.md` 放通用总规则
- 其他工具配置只做适配，不重新发明世界观

### 决策 2：正式建立指令栈

以后默认分成 5 层：

1. 仓库总规则
2. 工具专属规则
3. 路径 / 模块专属规则
4. 任务级 spec / plan
5. 运行时 hooks / 脚本检查

### 决策 3：正式建立三种执行模式

以后任何 AI 任务，先判断走哪种模式：

- 本地结对
- 后台代理
- 云端 PR 代理

### 决策 4：把 hooks 纳入主干

以后凡是“必须做”的动作，都尽量从提醒升级成自动检查。

优先级最高的 hooks 场景：

- 危险命令拦截
- 格式化 / lint / 测试
- 发布前证据检查

### 决策 5：把 Agent 从“角色”升级成“profile”

以后 agent 不能只写“负责什么”。

还要写清：

- 输入
- 输出
- 工具边界
- 完成标准
- 交接格式

---

## 三、这轮暂不吸收的 4 类内容

### 1. 各平台界面层的小功能

变化太快，不值得进母文档。

### 2. 没有审计和权限边界的全自动自治

风险太高。

### 3. 只有营销叙事、没有真实工作流证据的教程

信息密度低，容易带偏系统。

### 4. 只在某个单一工具里成立的硬编码流程

AIVibe OS 需要跨工具复用，不应被单一平台绑死。
