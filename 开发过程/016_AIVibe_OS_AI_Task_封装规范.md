# AIVibe OS AI Task 封装规范

## 目标

这份规范解决一个常见混乱：

AI 能力到底应该怎么对业务暴露，才能既好用，又不把模型细节漏到业务层。

答案不是“到处写 prompt”，而是把 AI 能力封成稳定的任务。

## 什么是 Task

Task 是业务层可直接调用的 AI 能力单元。

它应该表达的是“我要完成什么”，而不是“我要用哪个模型和哪段提示词”。

例如：

- `extractDesignTokens`
- `generateReleaseSummary`
- `classifyTicketIntent`

这些名字表达的是能力，不是实现细节。

## Task 必须包含什么

每个 Task 至少应有下面 6 个部分：

1. 输入定义
2. 输出定义
3. prompt 引用
4. 调用策略
5. 失败处理
6. 日志与评测挂点

## 输入输出原则

### 输入

输入必须清楚、最小、结构化。

不要把一大坨上下文无边界地塞进去。

### 输出

输出必须稳定、可消费、可校验。

优先返回结构化结果，不优先返回自由文本。

## 调用策略

每个 Task 在设计时都必须先回答：

- 默认走哪个 provider
- 有无备用 provider
- 允许多久超时
- 失败是否重试
- 是否允许走缓存

这些规则不应该散在业务层。

## 失败处理

Task 不能只有成功路径。

至少要提前定义：

- provider 报错怎么办
- 输出格式错了怎么办
- 输出质量不达标怎么办
- 超时怎么办

常见做法：

- 重试一次
- 切备用模型
- 返回保底结构
- 转人工或异步

## 命名规范

Task 名称用动词开头，表达业务能力。

推荐：

- `extractDesignTokens`
- `summarizeMeeting`
- `rewriteMarketingCopy`

不推荐：

- `callGpt4`
- `runPromptA`
- `llmHandler`

## 一个 Task 不该做什么

一个 Task 不应该：

- 处理多个不相关业务目标
- 混入 UI 逻辑
- 直接依赖某个页面组件
- 把 provider SDK 暴露给外层

如果一个 Task 已经太大，就拆成多个更小的任务。

## Task 的最小交付

一个新 Task 至少要同时具备：

- 结构化输入输出
- 对应 prompt
- 调用策略
- 日志字段
- 最小评测样本

没有这些，就不允许直接进生产链路。

## 人和 AI 的分工

AI 可以负责：

- 生成 Task 初稿
- 补 prompt 草稿
- 生成 schema 草稿
- 批量补测试样本

人必须拍板：

- 这个 Task 是否值得独立存在
- 输出结构是否稳定
- 失败策略是否合理
- 是否适合进入生产

## 一句话铁规矩

**Task 表达业务能力，不表达模型细节；业务只该依赖 Task，不该依赖 prompt 和 provider。**
