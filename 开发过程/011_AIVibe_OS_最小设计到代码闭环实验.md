# AIVibe OS 最小设计到代码闭环实验

## 目标

这份实验不是为了证明 AI 能把页面写出来，而是为了验证：

**AIVibe OS 设计到代码这条链路，是否真的比“直接看图写页面”更稳、更快、更可复用。**

## 实验范围

第一轮实验只做一个最小闭环，不追求大而全。

建议范围：

- 选择 1 个结构中等复杂的 Figma 页面
- 页面至少包含 3 到 5 个高频基础组件
- 页面最好包含列表、筛选、弹窗或卡片中的至少两类

不建议第一轮就选：

- 超复杂 dashboard
- 超长营销页
- 动画特别重的页面

## 实验目标

第一轮闭环要验证 4 件事：

1. token 能不能顺利进入代码链路
2. 基础组件能不能按映射规则复用
3. 页面能不能主要靠“拼组件和模块”完成
4. 最终结果是否比手工逐页还原更容易维护

## 实验步骤

### Step 1：选页面

选一个能代表常见业务界面的 Figma 页面，记录：

- 页面名称
- 页面类型
- 包含哪些基础组件
- 哪些地方可能是特例

### Step 2：抽 tokens

先从页面相关设计系统里抽出第一批 token：

- 颜色
- 字体
- 间距
- 圆角
- 阴影

输出目标：

- 一份结构化 token 数据
- 一份 token 命名草案

### Step 3：做组件映射

优先处理 3 到 5 个高频组件，例如：

- Button
- Input
- Card
- Modal
- Tabs

每个组件都要记录：

- Figma 名称
- 代码组件名
- 变体关系
- 缺失状态
- 是否需要扩展

### Step 4：拼页面

使用已有组件和 token，把页面主体拼出来。

目标不是把每个像素都手调到极致，而是验证：

- 页面主要依赖组件复用
- 页面没有大面积重新造轮子
- 页面层没有重新定义视觉系统

### Step 5：做结果评估

从 3 个角度评估：

- 还原效果
- 复用程度
- 后续维护成本

## 验收指标

这轮实验至少看下面 6 个指标：

1. 是否成功提取第一批 token
2. 是否至少映射 3 个高频组件
3. 页面是否主要由已映射组件拼出
4. 页面层是否显著减少硬编码视觉值
5. 是否能明确列出哪些地方值得沉淀成规范
6. 是否能明确列出链路中最卡的环节

## 成功标准

实验算成功，不要求完美还原，但至少要满足：

- token 链路跑通
- 组件映射规则可执行
- 页面复用比例明显高于“从零还原”
- 能明确提炼出下一轮该补的规则和模块

## 失败也算结果

如果实验失败，不代表方向错了。

真正要记录的是：

- 卡在 token 还是卡在组件映射
- 卡在 Figma 结构还是卡在代码骨架
- 是规则不清，还是工具不顺

这些信息本身就是 AIVibe OS 要吸收的资产。

## 实验产出

这轮实验结束后，至少要留下：

- 一份实验记录
- 一份问题清单
- 一份规则补丁
- 一份下一轮优化建议

## 一句话原则

**第一轮闭环的目标不是做出完美页面，而是验证这条链路值不值得成为 AIVibe OS 的默认做法。**
