# AIVibe OS 多代理执行模式：本地 / 后台 / 云端

## 这份文档解决什么问题

现在很多 AI coding 工具都支持：

- 当前对话里直接做
- 后台自己跑
- 云端自己改分支 / 提 PR

如果 AIVibe OS 不把这三种模式说清楚，结果就会很乱：

- 所有任务都挤在当前会话里
- 该隔离的任务没有隔离
- 该异步的任务还在同步等

这份文档的目标，就是给 AIVibe OS 固定一张分流表。

---

## 一、三种模式分别是什么

### 模式 1：本地结对

意思是：

- 人和 AI 在当前会话里一起推进
- 上下文最完整
- 决策来回最快

适合：

- 架构设计
- 高风险改动
- 需要频繁看代码上下文的任务
- 边做边定方案的任务

### 模式 2：后台代理

意思是：

- 任务先交出去
- AI 在后台自己跑
- 结果回来后再 review

适合：

- 长时间运行的任务
- 批量改动
- 不阻塞主线的 side task
- 大规模搜索、整理、生成

### 模式 3：云端 PR 代理

意思是：

- 任务在远端隔离环境执行
- 结果以分支、提交、PR 的形式回来

适合：

- 边界清楚的实现任务
- 能被独立 review 的改动
- 需要更强审计和权限隔离的任务

---

## 二、默认分流规则

以后 AIVibe OS 默认先按下面 4 个问题做判断。

### 1. 这件事风险高不高

- 高风险：本地结对
- 中低风险：可考虑后台或云端

### 2. 这件事边界清不清楚

- 边界不清：本地结对
- 边界清楚：后台或云端更合适

### 3. 这件事要不要占住当前上下文

- 必须来回讨论：本地结对
- 可以异步：后台或云端

### 4. 这件事需不需要更强隔离

- 需要分支、审计、权限边界：云端 PR 代理

---

## 三、最推荐的任务分配

### 优先留在本地结对的

- 产品 / 架构决策
- 高风险数据库改动
- 高风险发布动作
- 复杂 Figma 高保真还原
- 需要反复读上下文的疑难 bug

### 优先交给后台代理的

- 大规模文档整理
- 批量重构
- 搜索和比对
- 扫目录、补台账、补索引
- 低风险页面批量调整

### 优先交给云端 PR 代理的

- 独立功能实现
- 独立测试补全
- 独立规则文件整理
- 可被 reviewer 明确验收的改动

---

## 四、每种模式的最小交付要求

### 本地结对

至少留下：

- 本轮任务目标
- 改动说明
- 验证结果

### 后台代理

至少留下：

- 输入说明
- 输出结果
- 失败原因或未完成项
- 需要人工 review 的点

### 云端 PR 代理

至少留下：

- 分支 / PR
- 改动摘要
- 测试结果
- 风险说明

## 五、AIVibe OS 当前推荐工具映射

### 本地结对

- Claude Code
- Codex CLI
- Aider

### 后台代理

- Cursor Background Agents
- Codex app / 异步任务
- 本地子代理 / 多 agent 分治

### 云端 PR 代理

- GitHub Copilot coding agent
- GitHub custom agents
- Codex cloud / app 异步分支任务

---

## 六、最容易犯的错

### 1. 什么都留在本地

结果主上下文越来越重。

### 2. 边界不清的任务直接丢后台

结果回来一堆偏差。

### 3. 高风险任务直接交给云端

结果 review 成本暴涨。

### 4. 任务交出去了，但没有固定回收格式

最后还是得重新读一遍。

---

## 七、AIVibe OS 的推荐动作

以后交给代理前，先做一句判断：

**这件事是需要共同决策，还是只需要隔离执行。**

如果是前者，留在本地。

如果是后者，再从后台和云端里选。

---

## 一句话铁规矩

**不是所有任务都该在当前对话里做完；AIVibe OS 必须把本地、后台、云端三种执行模式正式分流。**
