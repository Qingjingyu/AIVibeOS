# AIVibe OS AI Evals 评测规范

## 目标

这份规范解决一个老问题：

AI 功能好不好，不能只靠“看起来还行”，必须有最小可重复的评测办法。

## 总原则

评测不是最后补的装饰，而是 AI Runtime 的一部分。

只要一个 AI Task 准备进入生产，就必须有对应的最小评测。

## 评测分三层

### 1. 样例回放

用固定输入反复跑，观察输出是否退化。

它解决的是：

- 改 prompt 后效果有没有变差
- 换 provider 后结构有没有变
- 重构 runtime 后结果有没有漂移

### 2. 规则检查

检查结果是否满足最基本要求。

例如：

- JSON 是否合法
- 字段是否齐全
- 长度是否超标
- 是否包含敏感内容
- 是否命中业务禁区

### 3. 人工抽检

不是所有问题都能靠规则看出来。

所以关键任务还要有人抽样判断：

- 结果是否真的有用
- 质量是否达到业务预期
- 是否存在明显胡说或误判

## 每个 Task 最低要有什么评测

每个准备进入生产的 Task，至少要有：

1. 一组固定样本
2. 一套规则检查
3. 一份人工抽检结论

少一个，都不算评测完整。

## 样本设计原则

样本不能只挑容易的。

至少要覆盖：

- 正常输入
- 边界输入
- 异常输入
- 容易误判的输入

如果一个 Task 只在最理想场景下表现好，那它还不够稳。

## 评测结果至少记录什么

- Task 名称
- 运行时间
- 使用的 provider 和模型版本
- prompt 版本
- 输入样本 ID
- 输出结果摘要
- 是否通过规则检查
- 人工结论

## 什么时候必须重新评测

下面这些情况发生时，至少要重新跑回放和规则检查：

- 改 prompt
- 换 provider
- 改输出 schema
- 改 runtime 调用策略
- 改缓存、重试或降级逻辑

## 评测和测试的关系

评测不是替代普通测试。

普通测试解决的是：

- 代码能不能正常运行
- 结构有没有被破坏

评测解决的是：

- AI 输出是不是依旧够好
- 功能有没有悄悄退化

两者都要有。

## 不允许的做法

- 只在第一次上线前人工看一眼
- prompt 改了却不回放
- 只测最理想样本
- 输出变差了但没有证据

## 一句话铁规矩

**没有样本回放、没有规则检查、没有人工抽检的 AI Task，不允许宣称“效果稳定”。**
