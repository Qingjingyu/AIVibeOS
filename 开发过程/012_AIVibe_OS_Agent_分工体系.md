# AIVibe OS Agent 分工体系

## 目标

这份文档解决一个根问题：

AI 不能再以“一个大模型全包”的方式工作，而要像一个小型产品团队那样明确分工、明确交接、明确责任。

## 总原则

AIVibe OS 默认采用多角色协同，不采用万能 Agent。

一句话：

- 人负责拍板
- Agent 负责执行
- 每个 Agent 只吃清晰输入，只交付清晰输出

## 角色清单

### 1. 产品 Agent

负责：

- 收敛需求
- 区分 MVP 和伪需求
- 整理 proposal / spec
- 明确验收标准

输入：

- 用户目标
- 业务背景
- 约束条件

输出：

- Proposal
- Spec 初稿
- 范围边界

### 2. 设计 Agent

负责：

- 读取 Figma
- 提取 tokens、组件信息、页面结构
- 判断哪些该复用，哪些该新增
- 输出设计到代码任务拆解

输入：

- Figma 页面或组件
- 现有组件库信息
- 映射规范

输出：

- tokens 草稿
- 组件映射草稿
- 页面模块拆解

### 3. 前端 Agent

负责：

- 实现页面、组件和交互
- 优先复用 `packages/ui`
- 接入真实接口或 mock 数据
- 补充前端测试和状态处理

输入：

- 设计拆解
- 组件映射结果
- 接口合同

输出：

- 前端实现
- 页面状态说明
- 前端测试结果

### 4. 后端 Agent

负责：

- 实现业务接口
- 设计数据流和状态流
- 处理权限、校验、存储和异步任务
- 输出接口与错误码说明

输入：

- Spec
- Design
- 接口需求

输出：

- API 实现
- 数据结构说明
- 日志与错误处理说明

### 5. QA Agent

负责：

- 按 spec 做测试
- 覆盖主流程、边界流程、异常流程
- 输出问题清单和回归结果

输入：

- Spec
- 验收标准
- 实现结果

输出：

- QA 记录
- 问题列表
- 回归结论

### 6. Review Agent

负责：

- 看架构是否清楚
- 看边界是否混乱
- 看是否有明显稳定性和可维护性风险
- 看是否真的达到发布条件

输入：

- 代码改动
- 测试结果
- 发布说明

输出：

- 风险清单
- 改进建议
- 是否建议继续发布

## 标准交接链路

默认按这条顺序协作：

`产品 Agent -> 设计 Agent -> 前端 / 后端 Agent -> QA Agent -> Review Agent -> 人类拍板`

复杂功能允许并行，但交接物必须先清楚，再开始下一步。

## 每个 Agent 的输出要求

所有 Agent 输出都必须满足：

- 边界清楚
- 可交接
- 可验证
- 能沉淀进 AIVibe OS

不允许只输出一堆零散想法。

## 不允许的工作方式

- 没有 spec 就让前端 Agent 开写
- 没有组件映射就让设计 Agent 直接还原整页
- 没有接口合同就让前后端互猜
- 没有 QA 和 Review 就宣布完成

## 人类负责什么

人不负责重复劳动，但必须负责关键判断：

- 定方向
- 定边界
- 定优先级
- 定是否验收通过
- 定是否上线

## 一句话铁规矩

**Agent 只负责把明确的事做快，不负责替人做最终判断。**
