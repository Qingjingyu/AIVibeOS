# AIVibe OS 项目启动模板入口

## 这份文档解决什么问题

现在新项目启动时，模板已经不少了：

- Research
- Proposal
- Spec
- Design
- Tasks

问题不是没有模板，而是第一次开项目的人容易分不清：

- 先用哪份
- 每份分别解决什么
- 哪份是主模板
- 写到什么程度才够进入下一步

一句话：

**这一页不是新模板，而是“项目启动模板组”的入口。**

## 二、每份模板分别负责什么

### 1. Research 模板

看：

- [01_Research模板.md](../02_Spec模板/01_Research模板.md)

它解决的是：

- 这次做什么方向
- 市场和对标情况怎样
- 有什么核心风险

它不是正式立项文档，而是输入。

### 2. Proposal 模板

看：

- [02_Proposal模板.md](../02_Spec模板/02_Proposal模板.md)

它解决的是：

- 为什么做
- 给谁做
- 这次只做什么
- 这次明确不做什么

它的核心任务是收范围。

### 3. Spec 模板

看：

- [01_产品需求Spec模板.md](../02_Spec模板/01_产品需求Spec模板.md)
- [06_如何写一个好Spec.md](../02_Spec模板/06_如何写一个好Spec.md)

它解决的是：

- 页面和流程
- 状态和交互
- 接口口径
- 验收标准

这是“能不能开工”的关键模板。

### 4. Design 模板

看：

- [02_技术设计Design模板.md](../02_Spec模板/02_技术设计Design模板.md)

它解决的是：

- 模块边界
- 数据流
- 外部依赖
- 技术方案

这是“怎么搭结构”的关键模板。

### 5. Tasks 模板

看：

- [05_Tasks模板.md](../02_Spec模板/05_Tasks模板.md)

它解决的是：

- 任务怎么拆
- 谁先做什么
- 怎么验证

这是从方案走向执行的桥。

## 三、谁是主模板，谁是辅助模板

### 主模板

真正要被项目直接复制使用的是：

1. `Research`
2. `Proposal`
3. `Spec`
4. `Design`
5. `Tasks`

### 辅助模板 / 辅助说明

用来帮助理解和提升质量的是：

- [00_模板使用说明.md](../02_Spec模板/00_模板使用说明.md)
- [06_如何写一个好Spec.md](../02_Spec模板/06_如何写一个好Spec.md)

这两份不是主模板，不要拿来代替项目文档本体。

---

## 四、什么时候可以进入下一份

### 从 Research 到 Proposal

当你已经知道：

- 大致要做什么方向
- 风险大概在哪

就可以进入 Proposal。

### 从 Proposal 到 Spec

当你已经把：

- 这次只做什么
- 这次不做什么

说清楚了，才能进入 Spec。

### 从 Spec 到 Design

当你已经把：

- 主流程
- 页面清单
- 核心状态

写清楚了，才能进入 Design。

### 从 Design 到 Tasks

当你已经把：

- 技术边界
- 外部依赖
- 模块拆分

定下来，才能拆 Tasks。

---

## 五、最容易犯的错

### 1. 跳过 Proposal

这样范围很容易失控。

### 2. 把 Design 当视觉稿

这里的 Design 先是技术设计，不是 UI 稿。

### 3. 还没写清 Spec 就拆 Tasks

这样任务会全是猜。

### 4. 把辅助说明文档当主模板

这样最后根本没有真正的项目文档。
