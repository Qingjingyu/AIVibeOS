# AIVibe OS DESIGN.md 视觉执行规范

## 这份规范解决什么问题

很多项目不是没有 `Spec`，而是 AI 在真正出页面时，缺少一份清楚的“视觉执行说明”。

结果就是：

- 结构可能对了，但气质跑偏
- 不同页面各写各的，风格不统一
- 没有 Figma 时，AI 只能靠猜
- 有 Figma 时，整体品牌感、缺失状态和非关键页面还是容易漂

`DESIGN.md` 就是为了解决这个空档。

它不是技术设计，也不是像素稿，而是一份给 Agent 看的视觉宪法。

## 它是什么

`DESIGN.md` 是项目根目录里的一个 Markdown 文件。

它用通俗语言写清楚：

- 这个产品看起来应该是什么气质
- 颜色、字体、留白、圆角、阴影大概怎么用
- 按钮、卡片、表单、导航这些关键组件应该长什么样
- 哪些做法可以接受，哪些不允许出现
- Agent 在生成页面时该优先追求什么

一句话：

**它负责告诉 AI“做成什么感觉”，而不负责告诉 AI“模块怎么拆、接口怎么接”。**

## 它不是什么

`DESIGN.md` 不能替代下面这些东西：

- 不能替代 `Proposal`
- 不能替代 `Spec`
- 不能替代技术 `Design`
- 不能替代高保真 Figma
- 不能替代组件映射和 token 编译链路

边界要分清：

- `Spec` 解决“做什么”
- 技术 `Design` 解决“怎么实现”
- `DESIGN.md` 解决“看起来和用起来应该是什么风格”
- Figma 解决“关键界面要精确到什么程度”

## 什么时候应该用

下面这些场景，默认建议补 `DESIGN.md`：

1. 没有 Figma，但要快速做出第一版高一致性界面
2. 有 Figma，但只覆盖核心页，其他页面容易风格漂移
3. 要做官网、营销页、品牌页、活动页
4. 要做早期原型，但又不希望看起来像临时 demo
5. 一个项目里会有多个 Agent 并行出页面，需要统一审美边界

## 什么时候单靠它不够

下面这些场景，`DESIGN.md` 只能做补充，不能单独顶上：

- 要求像素级还原的核心业务页面
- 组件状态非常复杂的企业后台
- 强依赖品牌细节的成熟设计系统
- 需要严格复用现有设计库的多端产品

这类场景还是要走：

`Figma -> tokens / 组件映射 / 页面模块 -> 实现`

## AIVibe 里的定位

在 AIVibe OS 里，`DESIGN.md` 的定位是：

**视觉执行约束层。**

它位于：

`Proposal / Spec / 技术 Design` 和 `真实页面实现` 之间。

可以理解成这条链路：

`问题定义 -> Spec -> 技术方案 -> DESIGN.md -> 组件/页面实现 -> QA`

这样做的好处是：

- 先把产品边界讲清楚
- 再把技术边界讲清楚
- 最后把视觉边界讲清楚

AI 就不容易边写边乱猜。

## 推荐文件位置

默认放在项目根目录：

```text
/DESIGN.md
```

v1 不建议把它拆成很多文件。

先保证：

- 所有人都知道去哪看
- 所有 Agent 都读同一份规则
- QA 和 Review 能对照同一份标准验收

如果未来一个项目存在多个品牌、多个子产品，再考虑扩展成：

```text
/design/brand-a.md
/design/brand-b.md
```

但这不是 v1 的默认做法。

## 标准结构

一份可用的 `DESIGN.md`，默认至少包含下面 8 块。

### 1. 产品气质

写清楚这个产品想给人的感受。

不要只写“高级、科技、现代”这种空话，要写成能执行的话，比如：

- 更像专业工具，不像营销海报
- 信息密度偏高，但不能压迫
- 动效要克制，重点是稳定和可信

### 2. 色彩与 Token

写清楚主色、辅色、背景色、边框色、成功/警告/错误色怎么分工。

如果项目已经有 token，优先写 token 名，不要到处写裸色值。

### 3. 字体与层级

写清楚：

- 标题和正文的层级关系
- 重点数字、说明文字、标签文字怎么区分
- 行高、字重、密度的大致倾向

### 4. 关键组件风格

至少覆盖：

- Button
- Input / Select / Textarea
- Card
- Table / List
- Modal / Drawer
- Navbar / Sidebar

重点不是把所有像素写死，而是写清楚这些组件“应该像哪一类产品”。

### 5. 布局与留白

写清楚：

- 页面宽度倾向
- 区块之间的疏密关系
- 栅格或模块排布习惯
- 手机端收缩时优先保留什么

### 6. 状态与动效

至少补齐：

- Loading
- Empty
- Error
- Disabled
- Hover / Focus / Active

如果没有这块，AI 很容易只做成功态。

### 7. Do / Don't

这是最值钱的一块。

直接写明：

- 可以出现什么
- 不可以出现什么

比如：

- 可以有柔和阴影，但不要大面积悬浮玻璃感
- 可以用渐变做强调，但不要把整个页面做成炫技背景
- 不要出现和品牌无关的随机插画风格

### 8. Agent 执行提示

最后单独留一段给 Agent：

- 生成页面时优先遵守什么
- 遇到冲突时谁优先
- 没写清楚时应该保守处理还是主动延展

## 写作规则

为了让 `DESIGN.md` 真能被用起来，写的时候遵守这 6 条：

1. 说人话，不堆设计黑话
2. 尽量写成可执行句子，不写空形容词
3. 先写必须遵守的，再写可选加分项
4. 优先写规律，不优先写零散特例
5. 能绑定 token 就不要只写颜色印象
6. 能写 Do / Don't，就不要只写“自由发挥”

## 最小模板

下面是一份可以直接复制到项目里的最小版本：

```md
# DESIGN.md

## 1. 产品气质
- 这是一个偏专业、可信、克制的产品，不是热闹型营销站。
- 优先追求清晰和稳定，不追求炫技效果。

## 2. 色彩与 Token
- 主色用于关键操作和强调，不要满页面滥用。
- 背景以中性色为主，保证内容可读性。
- 成功、警告、错误状态必须区分清楚。

## 3. 字体与层级
- 标题要明确强于正文，但不要过度夸张。
- 说明文字应清楚可读，不要为了“高级感”牺牲信息密度。

## 4. 关键组件风格
- Button 要明确、稳定、易点，不要做成漂浮玩具。
- Card 用于分组信息，层级清楚，避免无意义装饰。
- 表单优先强调清晰反馈和错误提示。

## 5. 布局与留白
- 页面保持稳定网格和清楚分区。
- 桌面端重视信息分组，移动端优先保留主路径。

## 6. 状态与动效
- Loading、Empty、Error、Disabled 都必须有对应处理。
- 动效只做辅助说明，不喧宾夺主。

## 7. Do / Don't
- Do: 保持统一圆角、间距和阴影语言。
- Do: 优先复用现有组件和 token。
- Don't: 随机混用多种视觉风格。
- Don't: 在没有依据时擅自加入重装饰效果。

## 8. 给 Agent 的执行提示
- 如果 Spec、技术 Design、DESIGN.md 冲突，先确认产品边界，再处理视觉细节。
- 如果页面没有明确参考，优先保守、统一、可复用，不要擅自炫技。
```

## 怎么接进 AIVibe 标准流程

推荐接法如下：

### 在 Proposal 阶段

先判断这次是否需要单独维护 `DESIGN.md`。

如果这个项目：

- 很看重品牌气质
- 没有完整 Figma
- 会同时出很多页面

那就直接立项时把它列进交付物。

### 在 Spec 阶段

Spec 里要写清楚：

- 哪些页面要遵守 `DESIGN.md`
- 哪些状态必须覆盖
- 有没有已有设计参考

### 在技术 Design 阶段

技术 Design 要补一句：

- 页面实现是否依赖 `DESIGN.md`
- 它和 token、组件库、Figma 的关系是什么

### 在 Tasks 阶段

任务里要明确：

- 谁来起草 `DESIGN.md`
- 谁来确认
- 哪些页面开发前必须先读它

### 在 QA / Review 阶段

除了看功能对不对，还要看：

- 视觉风格是否统一
- 状态页是否跑偏
- 是否出现明显违背 `Do / Don't` 的情况

## 跟 Figma 的关系

最简单的判断方式是：

- 有 Figma 时，`DESIGN.md` 负责补“整体气质”和“缺口约束”
- 没 Figma 时，`DESIGN.md` 负责给 AI 一个稳定的视觉方向

所以它不是 Figma 的对手，而是 Figma 的补充。

## 这轮吸收决策

基于 `awesome-design-md` 这类实践，AIVibe OS 先吸收下面 4 条：

1. 以后对没有完整 Figma 的项目，允许增加根目录 `DESIGN.md` 作为视觉约束文件
2. `DESIGN.md` 默认采用 Markdown，不另造复杂配置格式
3. `DESIGN.md` 默认服务于 Agent 执行，不写成纯设计师自嗨文案
4. `DESIGN.md` 必须和 `Spec`、技术 `Design`、QA 一起使用，不能单独宣称足够
